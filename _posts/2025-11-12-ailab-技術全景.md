21 個微服務，Python / Go / TypeScript 三種語言，怎麼選？

iKala AI Lab 從 2021 到 2025 年建了 21 個 ML 服務。搜尋引擎、推薦系統、文本分類、電腦視覺、NLP 基礎設施、LLM 標註平台。這篇不是每個服務的技術文件，是退一步看：21 個服務的技術選型背後，有沒有一致的決策邏輯？

我認為有。而且可以歸納成五種模式。

## 三層架構


先看全局。AI Lab 的技術棧分三層：

**應用層**：KOL Radar 的搜尋、推薦、標籤分類、業配偵測、輿情分析。這層面對用戶，延遲敏感，任何模型的改動最終都要反映在這裡。

**AI 服務層**：Hermes（搜尋引擎）、GeneralTag（主題分類）、TextAction（行為萃取）、TextSponsor（業配偵測）、TextKeyphrase（關鍵字萃取）、Autoprofile（人口統計預測）、CVFactory（電腦視覺）、NLPFactory（LLM 標註平台）、Social Graph（社群圖譜）。每個服務解決一個明確的問題。

**基礎設施層**：Elasticsearch、BigQuery、Neo4j、GCS、ONNX Runtime。5 台 GPU 機器（A6000 x16、RTX 6000 Ada x8、RTX 3090 x4），加上 Azure OpenAI、Google Gemini、Anthropic API。

21 個服務之間有明確的依賴關係。TextLangID 是所有多語言服務的入口，做完語言偵測之後才分流到其他 NLP 服務。NLP Infra（NER、POS、分詞）是 Autotag、Keyphrase、Hermes 的共用基礎。TextSponsor 的業配偵測結果是 ALS 推薦系統的訓練資料來源。NLPFactory 是所有模型的「標註工廠」——用 GPT-4 做 few-shot 標註，再拿標註結果微調小模型，最後打包成 ONNX 部署。

## 五種選型模式

### 先簡後深

每個服務的第一版都選最簡單的方案。Autotag v1 用 fastText，一個 n-gram 分類器搞定 27 個類別。效果夠用之後，v2 才換深度學習。到了 2026 年，General Tag 又從傳統 ML 全面轉向 LLM 標註。

TextSponsor 也是一樣，fastText 做業配文偵測，多語言版本（ja / vi / en / th / ms / zh）全都用 fastText。因為業配文的特徵很明確——有優惠碼、@ 品牌帳號、官網網址——簡單模型就能抓到。

我認為這是最重要的一條原則。很多團隊第一版就想上 transformer，但 fastText 能解決的問題不需要 BERT。先跑起來，有數據之後再決定要不要升級。

### 先通用後特化

Hermes 搜尋引擎的 embedding 模型選型是一個好例子。OpenAI 的 `text-embedding-3-small` 是通用 embedding，在 64 維和 256 維上都測過。但 Hermes 自己訓練的 MiniLM 64 維模型，在社群媒體文本的 NDCG 上表現更好。

因為社群媒體的文本太特殊了——emoji 大量出現、hashtag 污染、中英日混雜、語句不完整。通用 embedding 處理不好這些邊緣情境，domain-specific fine-tune 的效果差距很大。

推論速度也差很多：MiniLM 64 維（ONNX）的 model.run 只要 4.9ms，而 mcontriever（768 維）要 9.7ms，慢了一倍。64 維的壓縮是存儲、速度和品質之間的 trade-off，在社群文本這個 domain 上，64 維已經夠用。

### 先單語後多語

幾乎每個 NLP 服務都遵循同一條擴展路徑：zh → zh + ja → zh + ja + en → SEA 全覆蓋（vi / th / ms / ko）。

原因很實際：台灣是主力市場，先做中文。日本是第二大市場，接著做日文。英文是因為很多 KOL 混用中英文。SEA 是後來市場擴展才加的。

每次加新語言都要處理 ES analyzer 的問題。中文用 ik_max_word / ik_smart，日文要 kuromoji tokenizer + ngram，韓文用 nori tokenizer，泰文用 thai tokenizer，越南文要裝額外的 plugin。多語言不只是模型的問題，搜尋基礎設施也要跟著改。

### 先規則後 AI

Hermes 的演進路徑：純 BM25 → BM25 + Sentence Embedding 混合檢索 → RRF（Reciprocal Rank Fusion）融合。第一版只靠 ES 的 multi_match，效果夠用就先上線。有了用戶行為數據之後，才加入向量搜尋做混合檢索。

TextKnowledge 也是。先用 TextSell 做一個輕量的二分類器過濾——判斷這篇貼文是不是銷售文。只有正樣本才進入完整的知識萃取 pipeline。這個 pre-filter 省了大量推論成本。

### 先離線後即時

NLPFactory 的標準產出路徑：GPT-4 few-shot 標註（離線，$500-2000 / 任務）→ 小模型微調（離線，1-2 天）→ ONNX 打包部署（即時推論）。整個 pipeline 大約 8 天，比人工標註快 3.5 倍。

標註策略也有講究：KMeans 分 10 群，每群最多 2000 篇，確保 topic diversity。100 筆人工驗收就夠了。GPT-4 的 agreement rate 比 Claude v1.3 高，所以標註工廠一直用 GPT-4。

## 自研模型生態


AI Lab 自研了兩套基底模型。

**DiffCSE**：自研的社群媒體 domain encoder，有 60M（多語言通用）和 18M（英文輕量）兩個版本。Autotag、GeneralTag、TextSponsor、TextKeyphrase 都用這套 encoder。

**asym-t5-small**：自研的 encoder-decoder 模型。動機是 mbart 太大（610M），60M 的模型也被嫌大。目標是 128M 以下。asym-t5-small 在 benchmark 上跑贏 mt5-small-trim 和 google/t5-v1_1-small，推論速度最快（長文 512 seq 優勢明顯），記憶體用量最小。預訓練資料混合了 ikala/datahub-2022H2、Wikipedia、ThePile、zh_TW_c4。

所有 NLPFactory 產出的模型都走 ONNX 格式。GCS 上存了 11 組 ONNX 模型，每組一個 `model.onnx` 權重檔，上線前用 hashlib.md5 驗證完整性。

## 部署模式


不同模型用不同的部署模式：

- ONNX Runtime（CPU）：輕量推論用。Hermes encoder、Autoprofile CV、general_tag 都走這條路。
- PyTorch（GPU）：生成式模型。textaction、textkeyphrase（mt5-base）需要 GPU。
- FastText：分類任務。autotag v1.5、textsponsor、textlangid。FastText 的好處是完全不需要 GPU。
- Elasticsearch native：BM25 + dense_vector kNN，搜尋直接在 ES 裡做，不需要外部服務。

## 技術債與演進

21 個服務裡，4 個還在活躍迭代（TextAction、Search Quality、NLPFactory、CVFactory），17 個已經完成或歸檔。但「歸檔」不等於「不在用」——大部分模型以 ONNX 格式持續在線上服務，只是不再主動迭代。

2025 年底的一個重大決策：General Tag 從傳統 ML 全面轉向 LLM 標註。原因是品質不對稱——Production 模型的預測品質遠低於 LLM，而且新資料進入時，舊模型的低品質預測會覆蓋 LLM 產出的高品質標籤。維護成本太高，不如直接用 LLM 做線上標註。

這個決策連帶影響硬體：原本的 RTX 3090 x2 不夠用，需要升級到 RTX 6000 Ada x8。LLM 標註的 workload 排到了 2029 年——每個模型 x 每個平台 x 每個市場都需要獨立 labeling。

---

21 個服務的演進，最大的 pattern 是「漸進式複雜化」。沒有一個服務的第一版用最複雜的技術。每次升級都有數據驅動：品質不夠才換模型、延遲太高才做壓縮、新市場才加新語言。

但我一直在想一個問題：當 LLM 的推論成本持續下降，還需要維護這麼多「小模型」嗎？NLPFactory 的核心邏輯就是「用大模型標資料訓小模型降成本」，但如果大模型的推論成本降到跟小模型差不多呢？這 21 個服務裡有多少會被一個 LLM API call 取代？
