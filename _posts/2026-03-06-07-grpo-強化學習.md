最近在研究 DeepSeek R1 的訓練方法，才第一次認真搞懂 GRPO 是什麼。這篇是我的學習過程筆記，把我卡住過的地方、理解的邏輯都記下來。

---

一開始我以為 RLHF 就是 PPO，後來才發現 PPO 有個很實際的問題：它需要同時跑四個模型 — policy model、reference model、reward model、critic model（value function）。Critic 的參數量通常跟 policy model 一樣大，光是把這四個模型全載入記憶體就是噩夢。7B policy + 7B critic，再加上另外兩個，大部分團隊根本跑不動。

這個問題讓我理解了為什麼 GRPO 值得被提出來。

---

GRPO（Group Relative Policy Optimization）是 DeepSeek 團隊做的，核心想法讀完之後覺得很直覺：把 critic model 整個拿掉，改用「群組內相對獎勵」來估計 advantage。

我理解的是，做法是這樣的：對每個 prompt，從當前策略採樣一組（group）G 個回應，然後對這批回應的 reward 做 z-score 正規化：

```
Advantage_i = (reward_i - mean(rewards)) / std(rewards)
```

比組內平均好的回應得到正的 advantage（被鼓勵往這方向學），比平均差的得到負的（被壓制）。就這樣，不需要另一個模型來預測 value。

這邊卡了一下後來搞懂的是：advantage 的概念本來是「這個動作比我預期的好多少」，PPO 用 critic 來建立這個「預期」，GRPO 則是直接用同一批回應的平均來當基準。好處是省掉 critic，壞處是這個基準本身有一定的隨機性，所以 group size 不能太小。

目標函數沿用 PPO 的 clipping 機制（限制策略每步更新的幅度），加上 KL 散度懲罰防止偏離 reference model 太遠。和 PPO 一個微妙的差異是 KL penalty 的位置放法不同，GRPO 是直接從 surrogate loss 裡扣，不是加進 per-token reward。

---

看完論文後發現，GRPO 最讓我震撼的應用是 DeepSeek-R1 的訓練結果，特別是 R1-Zero 這個版本 — 不做任何 SFT，純用 GRPO，模型就能自發學出複雜的推理行為。

整個 R1 的訓練分四個階段：

第一階段，**冷啟動**：收集少量高品質長 CoT 資料先做 SFT，讓模型學會「怎麼展開思考過程」的格式，是後面 RL 訓練的起點。

第二階段，**推理 RL**：這才是 GRPO 真正發揮的地方。針對數學、程式、邏輯等可驗證任務做 RL，獎勵信號就只有「最終答案對不對」，完全不對推理過程施加任何約束。

第三階段，**拒絕採樣**：用 RL checkpoint 生成大量回應，篩選高品質的再做一輪 SFT。這步像是把 RL 學到的東西「蒸餾」固化。

第四階段，**全場景 RL**：結合 helpfulness 和 harmlessness 的獎勵信號，對所有任務類型做最終訓練。

DeepSeek 團隊說他們在訓練過程中觀察到「Aha Moment」— 模型開始會退一步、自我檢查、發現錯誤時主動修正方向。AIME 2024 benchmark 從 15.6% 跳到 77.9%。

我一直在想的是：這個推理能力不是被人教出來的，是在正確的獎勵信號下「湧現」出來的。這個結果讓我對 reward function 的設計重要性有了更深的體感。

---

說到 reward function，這是 GRPO 最核心的設計決策，也是最容易出問題的地方。

如果獎勵信號區分度不夠——同一個 prompt 的 G 個回應 reward 都差不多——advantage 就趨近零，模型學不到任何東西。這個問題在 PPO 裡不明顯，但 GRPO 裡特別敏感。

我整理了幾種 reward 類型的理解：

**Rule-based**：用程式邏輯驗證。數學題看最終答案、程式碼跑測試、JSON schema 驗證格式。R1 就是只靠這個訓練成功的，信號最乾淨，不會有 reward model 本身的偏差問題。

**Model-based**：用訓練好的 reward model 評分。適合沒有明確正確答案的任務，但 reward model 的偏差是一個隱藏的風險來源。

**ORM vs PRM**：ORM 只評估最終結果，PRM 評估每個推理步驟。這邊有一個我看到但還沒完全消化的說法：有研究指出 GRPO 在數學任務上等價於一個以 on-policy Monte Carlo 估計作為 process reward 的 RL 算法，也就是說 GRPO 其實隱含了 PRM 的特性。這個等價關係還在消化中。

數學和程式碼用 rule-based ORM 就夠了，通用推理可以用強 LLM 當 judge，但 judge 本身的偏差要小心。

把 GRPO 放在 RLHF 演算法譜系裡比較才看出各自的定位。

PPO 理論成熟、通用性強，但記憶體高、調參困難。DPO 實作最簡單、不需要 RL，但被靜態偏好資料的品質限制住，難以超越訓練資料。GRPO 記憶體低、訓練穩定，但需要大 batch 和有足夠區分度的可比較獎勵信號。KTO 只需要二元標註（thumbs up/down），標註成本低但信號弱。REINFORCE++ 用全局正規化取代 GRPO 的 group 正規化，社群用的人還比較少。

我理解的 2026 年 post-training 標準技術棧大概是：SFT 做指令跟隨 → DPO/SimPO/KTO 做偏好對齊 → GRPO/DAPO 做推理能力強化。GRPO 主要用在 RLVR（RL with Verifiable Rewards），PPO 主要用在傳統 RLHF。

---

超參數這塊是我覺得最容易踩坑的地方，記一下幾個關鍵數字：

7B 模型用 4-8 張 A100/H100 80GB 就能跑 GRPO，32B 以上建議 16-32 張 GPU 搭配 DeepSpeed ZeRO-3 或 FSDP。

Group size G 建議 16-64，DeepSeek 用 64。太小的話 advantage 估計變異會很大，等於基準不穩。KL 係數 beta 建議 0.001-0.04，過大限制探索，過小策略會崩壞，有研究說推理任務可以完全不用 KL。Learning rate 用 1e-6 到 5e-6，比 SFT 小很多，是為了避免策略劇烈變動。Batch size 建議 512-1024（prompts × group size），大 batch 是 GRPO 穩定訓練的必要條件。

常見失敗模式我看到幾個：

**Entropy collapse**：模型過早收斂到單一回應模式，只會生成一種答案。解法是降 learning rate、增大 group size，或用 DAPO 的 Clip-Higher 機制。

**Reward hacking**：模型找到獎勵函數的漏洞。比如格式獎勵太寬鬆，模型學會生成格式完美但內容空洞的回應。增加獎勵信號的多樣性可以緩解。

**KL 爆炸**：策略偏離 reference model 太遠。增大 beta 或降 learning rate。

**長回應退化**：CoT 越來越長但品質下降。這個我還沒親眼看到，但 DAPO 的 Overlong Reward Shaping 是針對這個問題提出的。

框架的話，TRL（Hugging Face）最好上手，生態整合最好。OpenRLHF 用 Ray + vLLM 的分散式架構，速度是 TRL 的 3.1 倍。verl（ByteDance）超參數最豐富（200+），原生支援 VLM 的 GRPO。Unsloth 記憶體優化極致，適合消費級 GPU。

---

最近的演進也值得追蹤。

**DAPO**（ByteDance）：針對長 CoT 訓練不穩定問題，提出四項改進 — Clip-Higher、Dynamic Sampling、Token-level Policy Gradient Loss、Overlong Reward Shaping。用 Qwen2.5-32B 在 AIME 2024 達到 50 分，只需要 R1-Zero 一半的訓練步數。

**Dr. GRPO**（GRPO Done Right）：指出原始 GRPO 目標函數存在偏差，移除標準差正規化項後理論上更正確。這個還沒深讀。

**Posterior-GRPO（P-GRPO）**：條件式過程獎勵，只對成功通過測試的回應施加 process reward，防止 reward hacking。

三個我覺得值得繼續追的方向：GRPO 的 off-policy 特性（有研究說它在實務上是 off-policy 的，跟原始論文的 on-policy 假設有出入）、Agentic RL（把 GRPO 擴展到工具使用和多步驟場景）、多模態 GRPO（verl 和 SWIFT 已支援 VLM 的 GRPO 訓練）。

---

還沒搞懂的部分：

1. GRPO 在數學上等價於以 on-policy Monte Carlo 估計作為 process reward 的 PRM 算法 — 這個等價性的推導我沒有跟完，論文裡應該有詳細推導，之後要回去讀。

2. GRPO 的 off-policy 特性：有說法指出實務訓練中 GRPO 其實是 off-policy 的，這個和理論上的 on-policy 假設怎麼對應，目前還不清楚。

3. Reward function 設計的邊界：什麼樣的任務「算是」有 verifiable reward？Data agent 的輸出格式算嗎？KOL 推薦文的品質呢？多維度 reward 的權重怎麼設？這個問題可能比學 GRPO 本身更難。
