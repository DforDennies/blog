老闆問 Paid Social 值不值得。我打開 GA4，97% Key Event 是 scroll。

不是誇張。GA4 的 Key Events（以前叫 Conversion Events）裡面，`scroll` 事件佔了 97%。所有管道的轉換率都被灌到 50% 以上，看起來每個管道表現都很好。但這個「好」是假的——只要用戶往下滑一點就算一次 Key Event。

GA4 裡面沒辦法取消 `scroll` 的 Key Event 標記。這個設定是之前某個人開的，現在改不了。所以第一個決定：在所有分析和報告中，手動排除 `scroll`、`[Amplitude] Page Viewed`（Amplitude 同步過來的事件）、`audience_organic_visitors`（Audience trigger）。


排除之後，真正的轉換率長這樣：

| 管道 | 修正前 Conv Rate | 修正後 Conv Rate |
|------|-----------------|-----------------|
| Email | 52.5% | ~18.8% |
| Organic Social | 51.2% | ~12.0% |
| Paid Search | 43.4% | ~6.2% |
| Direct | 52.2% | ~5.3% |
| Paid Social | 3.0% | ~0.3% |
| Display | 1.2% | ~0.1% |

Paid Social 的真實轉換率是 0.3%。Display 是 0.1%。這兩個管道加起來每月花的廣告預算，帶來的有效轉換趨近於零。

動手之前，先釐清 GA4 跟 Amplitude 各自的角色。兩個工具不該重複做一樣的事：

- **GA4 = 外部流量歸因引擎**——用戶從哪裡來？哪個管道帶來轉換？Paid 跟 Organic 的投報率？
- **Amplitude = 產品內行為分析**——用戶進來之後做了什麼？搜尋了什麼？收藏了哪些 KOL？

KolRadar 主站月活 39K（GA4 Property ID: 249781922），另外有一個日本部落格站 1.6K 週活。分析聚焦在主站。

**漏斗上層**：
- `View Signup Screen`（~6,149/月）——看到註冊頁面，代表有興趣
- `Login Success`（~3,085/月）——回訪登入，歸因哪個管道帶回舊用戶

**核心轉換**：
- `Account Created AD`（~629/月）——廣告主建立帳號，最重要的轉換
- `Account Created KOL`（~60/月）——網紅建立帳號
- `Clicks Register`（~581/月）——點擊註冊按鈕，用來量化註冊意圖

**高價值 Lead**：
- `Complete Premium Trial Form`（~208/月）——填寫試用表單，離付費最近的一步
- `Submit Contact-Us Success`（~35/月）——主動聯繫，高意圖客戶

**內容轉換**：
- `Download Success`（~43/月）——下載報告或白皮書

每個事件都有明確的漏斗位置和業務意義。不是「有人觸發了某個按鈕」，而是「這代表用戶走到了哪一步」。

## 管道健康評分

光看轉換率不夠。一個管道可能轉換率很高但量很小（像 Email，18.8% 但月流量只有 320），或者量很大但轉換差（像 Organic Search，47.5% 流量佔比但轉換率只有 2.1%）。

我設計了一個管道健康評分公式：

```
健康分 = (Conv Rate × 40%) + (Engagement Rate × 30%) + (增長率 × 20%) + (停留時間指數 × 10%)
```


跑出來的結果很直觀：

| 管道 | 健康評分 | 評級 |
|------|---------|------|
| Email | 85 | S |
| Organic Social | 80 | S |
| Paid Search | 75 | A |
| Direct | 70 | A |
| Referral | 55 | B |
| Organic Search | 55 | B |
| Paid Social | 15 | C |
| Display | 10 | C |

S 級的 Email 和 Organic Social 是被低估的管道。Email 月量只有 320，但轉換率是所有管道最高的，值得大幅加量。Organic Social 也是同理，5% 的流量佔比帶來 12% 的修正轉換率。

## 四個緊急問題

分析過程中挖出四個需要立即處理的問題：

1. **39% Landing Page 是 `(not set)`**——超過三分之一的流量找不到著陸頁。這代表追蹤設定有問題，GA4 無法判斷用戶第一個看到的是哪個頁面

2. **$190K 收入全歸 Unassigned**——GA4 裡有 19 萬美金的收入數據，但全部標記為 Unassigned 管道。歸因完全失效

3. **Direct 佔 30.6%**——Direct 流量理論上是用戶直接輸入網址或從書籤進來。但 30% 這個比例太高，通常代表很多流量因為缺少 UTM 參數被錯歸到 Direct

4. **SG +299%, TH +363% 異常增長**——新加坡和泰國的流量突然暴增，高度懷疑是 bot

## v2：BQ Export 全量分析

v1 用的是 GA4 API 和 UI 報表，但很快就碰到限制。GA4 UI 操作太慢、API 有 quota、維度組合受限。

v2 直接用 BigQuery Export。GA4 每天自動把完整事件資料匯入 BQ，從 2022 年 8 月開始已經累積了 2,126+ 張日表。這等於拿到了 raw data 的完全控制權。

v2 的核心改進：

**441 事件分 10 層**——比 Amplitude 的 486 事件更細的分層。L0 系統雜訊（17 個）、L1 流量互動（6 個）、L2 意圖和轉換（89 個）、L3 產品使用（287 個）、L4 付費（30 個）、L5 系統外部（12 個）。

**5 個漏斗**——流量漏斗、註冊漏斗、Campaign 漏斗、啟用漏斗、商業漏斗。其中註冊漏斗的數據最驚人：看到註冊頁的 118K 人，最終到付款的只有 266 人，整體轉換率 0.2%。

**跨站旅程**——KolRadar 有官網（`www.kolr.ai`）和 SaaS 應用（`app.kolr.ai`）兩個 hostname。用 GA4 的 user_pseudo_id 把兩邊的瀏覽行為串起來，才能看到完整的轉換路徑。月度平均 www → app 的跨站轉換率只有 1-12%，2026 年 1 月掉到 0.82%。

**Campaign 自動追蹤**——寫了 `track_campaign.py`，輸入 UTM campaign 名稱，自動從 BQ 拉數據、跑漏斗、發 Slack 通知。第一個追蹤的案例是 Connection Day 活動：86 人進站，只有 1 人完成試用啟用。

## Landing Page 洞察

v2 最有價值的發現之一是 Landing Page 的轉換率差異：

- `/influencer-discover/` 轉換率 **31.55%**——這是 KOL 探索頁，直接進入產品核心功能
- `/en/` 英文首頁 28.50%
- 中文首頁 `/` 只有 14.19%
- 大部分部落格文章 < 1%

部落格帶來不少流量，但幾乎不轉換。而 influencer-discover 這種功能導向的頁面轉換率是首頁的兩倍。這暗示行銷應該把更多預算放在功能頁的推廣，而不是內容行銷。

另外有個可疑發現：`/kol/@@@@_mock-1~5` 這五個假 KOL 頁面共有約 90K users，但 0% 轉換到 app。極可能是 QA 測試頁面被搜尋引擎收錄，或是爬蟲。

## 行動建議

分析跑完之後，我整理了五個方向。要說明的是，這些是從數據角度的觀察，不是我負責的領域，實際執行還需要行銷團隊判斷：

1. **檢視 Paid Social + Display 廣告效益**——0.3% 和 0.1% 的轉換率，數據上 ROI 偏低，但可能有品牌曝光等非直接轉換的目的
2. **Email 行銷值得關注**——轉換率 18.8% 是所有管道最高的，但月量只有 320。如果能擴大覆蓋，ROI 應該不錯
3. **修復追蹤**——移除 scroll Key Event 設定、修復 39% 的 Landing Page `(not set)` 問題，這是數據品質問題
4. **UTM 標準化**——所有行銷連結帶 UTM 參數，降低 Direct 30.6% 的虛胖比例
5. **高轉換 SEO 內容**——「網紅合約」關鍵字的文章轉換率 38.4%、「2026 趨勢」20.2%。找更多這種高轉換的搜尋意圖

---

我認為 GA4 最大的問題不是工具本身，而是「設定一次就不管了」的心態。scroll 被設成 Key Event 可能是兩年前某個人隨手開的，但這個設定讓之後所有的轉換率數據都變成垃圾。97% 的 Key Event 是 scroll，等於這兩年來所有看過轉換率報表的人，看到的都是假數字。

一個還在想的事：BQ Export 打開了很多可能性，但 GA4 UI 裡面的一些方便功能（像 Audience 和自動洞察）在 BQ 裡面得自己重建。哪些分析適合留在 GA4 UI，哪些該搬到 BQ，目前還沒有一個清楚的切分標準。
