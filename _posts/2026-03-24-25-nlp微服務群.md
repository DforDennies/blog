一篇貼文進來，要做語言偵測、關鍵字萃取、情緒分析、NER。四件事，四個模型，七種語言。這是 KolRadar AI Lab 的 NLP 服務群每天在做的事。

產品覆蓋台灣、日本、新加坡、印尼、泰國、越南、韓國七個市場。「多語言」不是 nice-to-have，而是 NLP pipeline 的設計基礎。每種語言的分詞方式不同、停用詞不同、模型表現差異大，一個模型打不了天下。

## 服務總覽


整個 NLP 文本處理服務群有九個核心服務：

1. **TextLangID** — 語言偵測（fastText），document-level + sentence-level
2. **TextKeyphrase** — 關鍵字/延伸詞萃取（seq2seq）
3. **TextKnowledge** — 品牌-商品知識抽取（SPDA）
4. **TextSponsor** — 業配偵測
5. **TextAction** — 行為萃取（六大類）
6. **NER** — 命名實體識別
7. **POS** — 詞性標註
8. **分詞** — 中文/日文分詞優化
9. **Sentiment** — 情感分析

所有服務統一用 ONNX 格式部署，模型存在 GCS（`ikala-ai-stg-ml-active/{service_name}/`），MD5 驗證確保模型完整性，版本管理一致（`v{major}.{minor}.{patch}-{date}`）。

## TextLangID：所有管線的起點

我認為 TextLangID 是整個 NLP 服務群裡最關鍵的一個，因為它是所有多語言服務的前置步驟：

```
TextLangID (語言偵測)
    ↓ 分流
    ├── zh → 中文 NLP pipeline（分詞 → NER → POS → TextKeyphrase → SPDA → GeneralTag）
    ├── ja → 日文 NLP pipeline（日文分詞 + POS → TextKeyphrase → GeneralTag）
    ├── en → 英文 NLP pipeline（TextKeyphrase → GeneralTag）
    └── vi/th/ms → SEA NLP pipeline（TextKeyphrase → GeneralTag）
```

偵測語言 → 決定用哪個語言的模型 → 過濾非目標語言的貼文 → 為 ml_datahub 更新 meta 欄位。目前是 v1.5.4，修過 emoji 導致的偵測錯誤，也研究過去除 hashtag 後語言偵測的準確度變化。

如果語言偵測錯了，後面整條 pipeline 都會跟著錯。所以它的精度要求最高，但維護成本相對低 — fastText 模型本身很穩定。

## TextKeyphrase：七語言的關鍵字萃取


TextKeyphrase 用 seq2seq 模型做關鍵字萃取和生成，基底模型是 `ikala-ray/mt5-small-trim`（121M 參數，是 mt5-small 的 trimmed 版本，減少了詞彙量以縮小模型大小）。

訓練資料用 GPT-4 透過 NLPFactory 標註，支援七種語言：中文、英文、日文、越南文、泰文、馬來文、韓文。

一個有趣的發現：中文的 F1 score < 0.3，其他語言都 > 0.3。原因不是模型差，而是 ground truth 的標準不同 — 中文由人類標註（較嚴格），其他語言由 GPT-4 標註。人類對「什麼算關鍵字」的判斷比 GPT-4 嚴格很多。

也比較過 mt5-small-trim（121M）和 bigscience/mt0-small（300M），差異不大，所以選了小的。長文本有 `do_chunk` 選項，根據語言和文本長度決定是否切分 chunk 再分別萃取。

越南文有個特殊問題：部分字符不在字典內（`ẩ`、`ẻ`、`ầ` 等），需要特殊處理 tokenizer。這種語言層面的 edge case，只有真的在多語系統上跑過才會碰到。

## TextKnowledge 與 TextSponsor

TextKnowledge 用 SPDA（Sponsor Post Detection & Attribution）方法，從貼文中抽取品牌名稱和商品名稱。目前以中文通用版本為主，處理品牌別名和簡稱的對應。

TextSponsor 用 FastText 做業配文偵測，已支援多語言。這兩個服務合在一起可以回答一個很有商業價值的問題：這篇貼文是不是業配？如果是，業配的是哪個品牌的什麼商品？

## NER 和分詞：語言差異最大的地方

NER（命名實體識別）和分詞是各語言差異最大的地方。

中文 NER 做了微調（`NLP improve zh NER finetuning`），用於抽取品牌名、人名、地名。日文分詞和 POS 需要特殊處理 — 用 `kuromoji_tokenizer` + `kuromoji_baseform`，另外加 ngram（2-gram）。

中文分詞優化有三個面向：
1. 後端模型部署（ray）
2. 詞彙挖掘 — 從社群語料發現新流行詞彙
3. 分詞結果品質評估

文本挖掘新詞的 POC 很有意思 — 社群媒體上的新詞（網路用語、品牌暱稱）不在標準字典裡，需要持續從語料中發現新詞並更新分詞字典，才能維持分詞品質。

情感分析用 BERT-based 模型做正/中/負三分類。日文情感分析另外做了特化版本。各語言的停用詞清單也需要持續維護，影響搜尋和分類品質。

## 資料管線架構


整條管線的流向：

社群媒體貼文 → TextLangID 分流 → 各語言 NLP pipeline → ml_datahub（meta 更新）→ BigQuery（結果寫入）→ 產品端（搜尋/推薦/分析）

每個語言的 pipeline 不完全一樣。中文最完整（分詞 → NER → POS → TextKeyphrase → TextKnowledge → GeneralTag），日文次之（日文分詞 + POS → TextKeyphrase → GeneralTag），英文和 SEA 語言最精簡（TextKeyphrase → GeneralTag）。

這個設計是根據各語言的資源和優先級決定的。中文是最成熟的市場，模型最完整。日文是第二大市場，有特化的分詞和情感分析。SEA 語言用共享的 pipeline，隨市場成長再逐步特化。

## 輔助工具

幾個值得一提的輔助工具：

- **Vocabulary Trimming**：減少模型詞彙量以縮小模型大小，ikala-ray/mt5-small-trim 就是這樣來的
- **Text Deduplicate Pipeline（d4）**：MinHash 大規模文本去重
- **Elasticsearch Autocorrect & Autocompletion**：搜尋自動校正和自動完成

所有實驗記錄在 WandB，模型訓練過程可追溯。

---

設計哲學三句話：語言偵測分流、語言特化 pipeline、統一部署標準。每個服務獨立部署、獨立更新，但共享同一套 ONNX + GCS + MD5 的部署規範。

我還在想的問題是：隨著 LLM 越來越強，傳統的 NLP pipeline（分詞 → NER → 情感分析）是不是該逐步被 LLM 取代？目前 production model quality < LLM quality 是已知的不對稱問題，但 LLM 推論成本高，每天數百萬篇貼文全過 LLM 不現實。蒸餾策略或許是折衷的方向，但具體怎麼做、哪些服務先替換，還沒有答案。
