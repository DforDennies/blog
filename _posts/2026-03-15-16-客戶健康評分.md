這是我推動內部數據自動化的其中一條分支。業務問我：「這個客戶下個月會不會續約？」我打開 CRM 看不出來。

CRM 上面寫著「合約中」、「年約」、「最後一次聯繫：三個月前」。回答不了那個問題。客戶有沒有在用產品？用了哪些功能？最近活躍度是上升還是下降？CRM 裡完全看不到。

KolRadar 是一個網紅行銷 SaaS，客戶付費後用來搜尋網紅、建立收藏庫、發商案、產報告。我手上有 Amplitude 的完整行為數據——8.9GB 的壓縮檔、144M 筆事件、27,898 個工作區。問題是這些數據散落在 18,724 個 JSON.gz 檔案裡，沒有人把它整理成業務能用的東西。

## 486 事件分 21 類

第一步是把事件分類。Amplitude 裡有 486 種不同的事件名稱，從 `Search for KOL` 到 `Click Pricing Plan` 到 `Stay KOL Detail`。業務不可能看 486 個事件，需要一個合理的分類。

我把這 486 個事件歸進 21 個類別，再按使用頻率分成四個群組：
- **主力功能**（佔比 > 1%）：搜尋、KOL 瀏覽、收藏、解鎖
- **常用功能**：商案管理、報告、篩選
- **偶用功能**：Chrome Extension、AI 搜尋、競品分析
- **未使用功能**：客戶從未觸發過的類別

這個分類有個反直覺的發現：一次搜尋意圖在 Amplitude 裡會觸發 5-12 個事件。用戶搜一次網紅，系統會記錄 `Search for KOL`、`View Search Result`、`Click Search Result`、`Stay KOL Detail` 等一系列事件。719K 的搜尋相關事件，實際搜尋意圖可能只有 80K-150K。

所以我又做了一層意圖級指標（Intent-Level Metrics），把膨脹的事件去重還原成 10 個真正有意義的動作：搜尋意圖、點擊結果、KOL 瀏覽、收藏、解鎖、提案、商案發送、商案完成、報告、競品分析。

整條 Pipeline 用 Python + SQLite 搭建，分三層：

**Layer 1：pipeline.py**——增量 ETL。把 18,724 個 `.json.gz` 檔案逐一解壓、解析、寫入 SQLite。用 `processed_files` 表追蹤已處理的檔案，支援增量更新。首次 ingest 跑大約 15 分鐘，886 秒處理完 144M 筆事件。

SQLite 裡的核心表包括 `daily_events`（日級聚合）、`daily_event_detail`（v2 事件明細）、`monthly_intents_v1` / `v2`（月級指標快取）。還有一組 v2 明細表追蹤搜尋品質、解鎖行為、收藏變化這些細粒度數據。

**Layer 2：data/{ws_id}.json**——每家公司一個 JSON 檔，包含 meta、months、totals、users、categories、intent 指標、v2 健康指標、付款紀錄、CRM 資料。27,913 個 JSON 檔案，其中 CRM 配對到的 464 家有完整 v2 指標。

**Layer 3：customer_dashboard_v2.html**——互動式儀表板。動態載入公司 JSON，Chart.js 畫圖表，支援多公司切換和日期範圍篩選。

整個架構沒有用 Airflow，沒有用 dbt，沒有用任何排程框架。就是一個 1,740 行的 `pipeline.py`，用 CLI 介面操作：`ingest` → `compute` → `export` → `ingest-crm`。

## 八維健康評分


每家公司的健康評分由八個維度加權計算，滿分 100：

- **搜尋活躍**（20%）——最近一段時間的搜尋頻率
- **漏斗深度**（15%）——從搜尋到收藏到商案的漏斗推進程度
- **搜尋品質**（15%）——ECTR（Effective Click-Through Rate），搜尋後有效點擊的比率
- **團隊規模**（10%）——活躍用戶數
- **穩定度**（10%）——活躍天數的穩定性
- **成長趨勢**（10%）——事件量的月度變化
- **功能採用**（10%）——21 類功能中使用了幾類
- **行為成熟**（10%）——工作流成熟度、重複解鎖率等進階行為

這個權重不是算出來的，是跟業務討論後拍的。搜尋佔最大權重，因為搜尋是用戶使用產品的第一步，搜尋活躍度下降通常是流失的先兆。

## 單一客戶 Dashboard

Dashboard 做到 v3 版本才穩定，中間有個插曲：用戶跟我說「上一版比較好看」，我差點理解成要 revert 到 v2，結果他說的「上一版」其實是 v3。後來他直接跟我說「不要再亂動了，先聽我說」。改 UI 之前一定要先確認用戶指的是哪個版本——這是踩出來的教訓。

最終的 v3 Dashboard 有三個主要 Tab：


**概覽**——健康評分 SVG 儀表盤（五維動態計算）、KPI 卡片、風險訊號、AM 行動建議。行動建議附帶 CTA 話術，業務可以直接複製貼到客戶訊息裡。

**功能分析**——21 類別按使用率分群，支援時間範圍篩選和多選比較。點進去還有二級事件 drill-down：選一個類別會顯示底下所有事件，分三組（≥5% / 1-5% / <1%），再點事件可以看月度趨勢。

**趨勢深度**——堆疊面積圖看類別組成變化、星期活躍模式、月份季節性指數。

## Benchmark：跟誰比？

單一客戶的數字本身沒有意義，需要基準線。我建了兩條 Benchmark：

1. **全體平均**——約 6,540 個活躍工作區（排除 21,366 個 < 1K events 的試用帳號和 13 個內部帳號）
2. **CRM 平均**——464 個有簽約紀錄的客戶

Benchmark 資料粒度做到日級（781 天），可以動態切月度、季度、年度。Dashboard 上用虛線疊在客戶的圖表上，紅線是全體平均，橙線是 CRM 平均。

ECTR 這類衍生比率不能直接從 benchmark.json 取，要在前端用 `bmRatio()` 算——用 click_result 的 benchmark 除以 search_intents 的 benchmark。這是因為 ratio 的平均不等於平均的 ratio。

## 從 HTML 到 Next.js

v2 做到後來，2,940 行的單一 HTML 檔案已經到了極限。全局總覽、多公司比較、篩選器交互作用，全部擠在一個檔案裡。Chart.js 實例超過 20 個，切 Tab 前要先 `destroyAllCharts()` 清理。

所以 v3 遷移到 Next.js 16 + Neon PostgreSQL + Vercel。架構完全重構：

- SQLite 的聚合數據推到 Neon PG（免費方案 512MB，剛好夠用）
- 每個 Tab 變成獨立的 route，Server Component 做首次渲染
- Benchmark、QoQ 動態、Top 20 都走 API route，支援時間範圍和國家篩選
- NextAuth v5 做 Google OAuth 白名單登入

遷移過程中 Neon 的 512MB 限制逼著我做了不少取捨——月度子事件只保留最近 6 個月（229MB → 87MB），KOL 互動明細做日級聚合而不是事件級。

## 漏斗修正

原始設計中，解鎖（Unlock）放在收藏（Collection）的下游，像是「搜尋 → 點擊 → KOL 瀏覽 → 收藏 → 解鎖」。但實際觀察數據發現，很多用戶是先解鎖再收藏，或只解鎖不收藏。

## GSheet 推送

最終產出除了 Dashboard，還有一份 Google Sheet，推了三個 Tab：
- 客戶 Persona 總覽（479 家）
- 客戶月度歷程（4,797 筆）
- 無 Amplitude 資料客戶（15 家）

業務可以直接在 Sheet 裡按合約到期日排序，看哪些客戶最近活躍度下降、哪些功能沒在用，提前準備續約話術。

---

我認為整個專案最值得的產出不是 Dashboard 本身，而是那套 21 類事件分類和 10 個意圖級指標。有了這層抽象，後面不管是做客戶分群、流失預測、還是功能採用分析，都有一個穩定的基礎。

不過八個維度的權重還是拍腦袋拍的。如果有足夠的流失標籤數據，用 logistic regression 或 survival analysis 去學權重，應該會更準。但這就要先回答另一個問題：怎麼定義「流失」？合約到期不續約算，那降級呢？從年約改月約呢？這些邊界案例，我還沒有一個滿意的定義。
