Prompt 寫好了，然後呢？從 prompt 到微調到上線，中間還有一大段路。

LLM 產品化不只是「接 API → 寫 prompt → 部署」。在 iKala AI Lab，我們走的是另一條路：大模型生成標註資料，訓練小模型，打包成 ONNX，線上做即時推論。路更長，但成本低兩個數量級。從 NLPFactory 到 Query Expansion Fine-tuning，這是完整的實戰記錄。

## 大模型標資料，小模型跑推論

2024 年初的 TGIF 內部分享有一組數字：大小模型的成本差 250 倍（速度）和 2.5 倍（金額）。這是 NLPFactory 存在的理由——不是用 LLM 直接上線服務，而是把 LLM 當「標註工廠」。

NLPFactory 的核心流程：

1. **Prompt 設計**：Few-shot + 要求模型生成解釋。這是最貴但最有效的方法。GPT-4 的 agreement rate 比 Claude v1.3 高，所以我們一直用 GPT-4。
2. **採樣策略**：KMeans 分 10 群，每群最多 2000 篇，確保 topic diversity。不是隨機抽，是有策略地覆蓋各種主題。
3. **人工驗收**：100 筆就夠了。不需要標幾千筆，因為 GPT-4 的輸出品質夠穩定。
4. **小模型微調**：拿標註結果訓一個 <1B 的模型。
5. **ONNX 打包**：微調完轉 ONNX 格式，上線做 CPU 推論。

整個流程大約 8 天（標註 3-5 天、微調 1-2 天、打包 1 天），比人工標註快 3.5 倍。每個任務花 $500-2000 USD。

TextAction、TextKeyphrase、TextKnowledge、General Tag——AI Lab 的核心 NLP 服務全都走這條路。

## 踩過的坑


**Mode collapse。** LLM 標註的一個常見問題：所有分類結果集中在少數幾個類別，輸出的文本形式也趨於一致。解法是改 prompt——加入更多 few-shot 範例，特別是邊緣案例的範例。

**GPT-3.5 拒絕輸出非英文。** 早期嘗試用 GPT-3.5 做中文標註，模型會直接拒絕或切換成英文輸出。後來統一改用 GPT-4。

**品質不對稱問題。** 這是 2025 年底 General Tag 全面轉向 LLM 的直接原因。Production 模型的預測品質遠低於 LLM，新資料進入時舊模型的低品質預測會覆蓋 LLM 產出的高品質標籤。傳統的「大模型標資料→小模型跑推論」模式在某些場景下開始失效，因為兩者的品質差距太大了。

## TextAction-En：資料品質流水線


TextAction 的英文版（TextAction-En）是一個很好的案例，展示了資料品質對模型效果的影響。

原始版本用 LLM 標註 101,913 筆英文資料，跑了大約 7 小時。但直接拿這批資料訓練，效果不夠好。後來加了兩個改進：

**加入空標。** 讓模型知道「這篇貼文不包含任何行為意圖」也是一個合法的輸出。這一個改動讓 accuracy 提升了 25%。因為原始標註資料裡幾乎沒有空標的範例，模型學到的是「每篇貼文都有行為意圖」，在實際推論時就會亂猜。

**Reward model 二次過濾。** 標完之後不直接拿去訓，先用一個 reward model 過濾品質差的標註。再加上新設計的資料選擇流水線，又提升了 10% accuracy。

部署上也做了優化：轉 ONNX Runtime 之後，推論速度從 15.4 秒 / 1000 筆降到 8.1 秒，快了 2 倍。訓練時用 Grid Search 跑 Cosine LR Scheduler + BERT Score 找最佳超參數。

## Query Expansion Fine-tuning

2026 年的新方向是 Query Expansion——教一個小模型（Qwen3-4B）學會搜尋查詢擴展。

使用者在 KOL 平台搜「台灣美妝 YouTuber」，模型要能擴展成三種形式：

- **lex**（Lexical）：美妝 KOL YouTube 台灣 → 給 BM25 全文檢索用
- **vec**（Vector）：在 YouTube 上活躍的台灣美妝 KOL → 給向量搜尋用
- **hyde**（HyDE）：模擬一筆理想搜尋結果的假想文件 → 用 embedding 反向搜尋

為什麼要 fine-tune 而不是直接用大模型？還是那個原因：太慢太貴。

Pipeline 五步：

1. 用 Azure OpenAI（GPT-4.1-mini）產生訓練資料，200 筆以上
2. 驗證資料品質，5 個維度打分
3. 格式轉換，切 train / eval
4. LoRA 訓練（Qwen3-4B，LoRA rank=16，3 個 epoch，學習率 2e-4，cosine scheduler）
5. Merge LoRA adapter 到基底模型，測試推理

LoRA 的好處是只訓練原始參數量的約 1%，一張 24GB VRAM 的 GPU 就能跑。合併後的模型約 8GB，部署時直接讀取，不需要再載基底模型。

這個 pipeline 的設計思路跟 NLPFactory 一脈相承：大模型（GPT-4.1-mini）生成高品質訓練資料，小模型（Qwen3-4B）學會同樣的能力，部署時只跑小模型。

## General Tag 的典範轉移


General Tag 是另一條演進路徑。從 fastText（2021）到深度學習 HMTC（2022-2024），再到 LLM 自動化標註（2025-2026）。

三年間的演進：

- **v0.1.0-alpha**：用 NeuralClassifier（Tencent 開源）做 HMTC，模型選型實驗跑了 HMCN、TextRCNN、TextRNN、DiffCSE 四種架構。HMCN 的 F-score 0.903 勝出。
- **v0.2.0**：模型從 60M（ikala-all-base-generator）縮到 18M（ikala-zh-small-generator），1000 筆推論大約 30 秒。轉 ONNX 之後在 CPU 上就能跑。
- **Phase 2（2025）**：發現語意誤判問題——周杰倫的貼文被標成「運動」（因為他代言運動品牌），男女朋友互稱「寶寶」被標成「親子」。這些邊緣案例靠傳統分類器很難修。
- **Phase 3（2026）**：全面轉向 LLM。用 gpt-oss 120B + Qwen3-VL 8B 做線上標註。台灣 FB 的 180M posts 已經標完 171M。動態 lvl1 / lvl2 tag 機制設計中。

我認為 General Tag 的演進就是整個 AI Lab 技術路線的縮影：先簡後深，先離線後即時，最終被 LLM 原生能力取代。

## 從 Prompt 到產品的完整路徑

把這些經驗串起來，LLM 產品化的路徑大概是這樣：

1. **Prompt 驗證**：先用大模型的 prompt 確認任務可行，測幾十筆看效果
2. **資料生成**：NLPFactory 模式，LLM 標註 + 採樣策略 + 人工驗收
3. **小模型訓練**：LoRA / SFT，模型大小控制在 128M 以下（如果不需要生成式能力）
4. **品質保證**：空標處理、reward model 過濾、Grid Search 超參數
5. **部署優化**：ONNX 轉換、MD5 驗證、CPU 推論
6. **監控迭代**：Model Performance Detection，月度 LLM-as-a-Judge 稽核

每個步驟都有可能卡住。Prompt 效果好不代表小模型能學會。小模型訓好不代表 ONNX 轉換沒問題。ONNX 跑得動不代表線上延遲可接受。

---

「大模型標資料訓小模型」在 iKala 跑了三年，從 NLPFactory 到 Query Expansion 一直有效。但 General Tag 的例子顯示，大小模型品質差距大到某個程度，這套邏輯就動搖了。

我在想的是：什麼時候「直接用 LLM 上線」會比「訓小模型」更划算？答案可能不是一個固定的時間點，而是每個任務的 cost-quality trade-off 各不相同。
