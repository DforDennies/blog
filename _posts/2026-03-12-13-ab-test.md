要同時跑 10 個實驗、每個流量不同、用戶不能重複中獎——這是 KOL Radar 內部 AB Testing 系統的設計起點。

大部分人談 AB Test 聚焦在統計學：p-value、信賴區間、樣本數計算。但在實作層面，**流量分配架構**才是最先碰到的問題。分流設計有缺陷，再精確的統計分析都沒意義。

這篇拆解我們設計 AB Testing 規格書的過程，包含架構選型、統計基礎、常見陷阱。

---

## 星球、世界、桶

<!-- 📸 建議截圖：Planet / Layer / Bucket 三層架構示意圖——星球內含多個世界，每個世界包含 Q 個桶的層級關係（無需脫敏） -->

我們的架構用了三層比喻：**星球（Planet）、世界（Layer）、桶（Bucket）**。

- **桶**：把用戶均勻分到 Q 個 bucket 裡。每個 bucket 是分流的最小單位
- **世界**：N 個桶組成一個世界（Layer N）。不同世界裡 Q 個桶的排序位置不同，用 shuffle 保證公平性
- **星球**：所有的世界看為一顆星球。星球有生命週期，到期後不能再增加實驗，等現有實驗結束後死亡，再生出全新的星球

在每個世界裡可以開實驗，按需要的流量劃分——50/50、40/60、隨你切。

### 為什麼用這個架構

核心概念是**分層正交（Layered Orthogonality）**：每一層用獨立的 hash 函數對用戶分桶，使得不同層的實驗在統計上互相獨立。同一個用戶可以同時參與多個層的實驗，不會互相干擾。

這跟 Google 2010 年在 KDD 發表的 Overlapping Experiment Infrastructure 是同一套思路：Domain → Layer → Experiment 的階層架構。每個 Layer 管控一組不可重疊的參數（例如 UI 參數和排序參數分屬不同 Layer），同一個請求最多同時參與 N 個實驗（每層一個）。

### 分流邏輯

線上用戶帶著 device_id 和 uuid 進來，系統先用 Bloom Filter 查詢：

- **已知用戶**：直接返回分桶結果
- **未知用戶**：從 count 較少的桶中隨機放入一個，更新計數

<!-- 📸 建議截圖：分流邏輯示意圖——device_id / uuid 進入 → Bloom Filter 查詢 → 已知/未知用戶的兩條路徑（無需脫敏） -->

Bloom Filter 在這裡的角色是用極小的空間快速判斷用戶是否已經分過桶。每個元素只需要約 9.6 bits 就能達到 1% 的假陽性率。假陽性（錯判為已知用戶）在流量分配場景影響極小，但可以大幅節省記憶體——對高 DAU 應用很關鍵。

如果是組合實驗（例如同時跑 Similar KOL + Search 的交叉實驗），key 會以 function + name 去設定。看指標就按 bucket 做 group by。

## 統計基礎：在開始之前就決定結局

### Power Analysis

任何實驗之前必須完成檢定力分析，決定所需樣本數。四個關鍵參數：

- **顯著水準 α**：慣例 0.05（95% 信心水準）
- **檢定力 1−β**：慣例 ≥ 0.80
- **MDE（最小可偵測效應）**：你希望能穩定偵測到的最小差異。MDE 越小，所需樣本越大
- **基礎指標變異數 σ²**：波動越大，需要越多樣本

每組所需樣本數近似為：`n ≥ 2 × ((Z_(1-α/2) + Z_(1-β)) / (Δ/σ))²`。以 α=0.05、Power=0.80 為例，n ≈ 16/d²（每組），其中 d 是 Cohen's d 效應量。

### 信賴區間比 p-value 更有用

p-value 只告訴你「有沒有顯著」，信賴區間告訴你「效應多大、不確定性多高」。實務上應該同時報告兩者，讓決策者了解效應的大小和範圍。

## 業界系統比較

<!-- 📸 建議截圖：四大業界 AB Test 系統比較表——Google / Facebook / Microsoft / Netflix 的核心特點對比（無需脫敏） -->

我們的架構不是憑空設計的。研究過四個業界系統：

**Google（2010）**：分層正交實驗架構的奠基論文，每層管控一組參數，每層獨立 hash seed，支援數千並行實驗。

**Facebook PlanOut**：強調實驗設計與應用程式碼分離，確定性分配（非真隨機），自動曝光日誌只在存取參數時觸發——這對統計檢定力很重要。

**Microsoft ExP**：四大元件（入口、執行、日誌、分析），最大貢獻是 SRM 偵測和 CUPED 變異數縮減。年逾萬實驗規模。

**Netflix**：排序演算法評估率先採用 Interleaving 方法和 Sequential Testing。Airbnb 後來用 Interleaving 達到約 50 倍加速（僅需 6% 流量、1/3 時間）。

我們的 Planet / Layer / Bucket 架構最接近 Google 的設計，但 multi-tenant 的 tenant 是星球，比較適合產品內部多團隊同時跑實驗的場景。

## 五個必知的陷阱

### 1. Peeking Problem（偷看問題）

在計劃的樣本數收集完之前反覆檢查結果，會嚴重膨脹 Type I Error。偷看 10 次，名義上 1% 的顯著水準實際上可能是 5%。

解法：用 Sequential Testing 或 Always-Valid P-values。如果真的需要中途監測，必須在統計框架內合法地做。

### 2. SRM（Sample Ratio Mismatch）

實際分組比例跟預期不符。用 Chi-squared 檢定偵測，p < 0.01 就該告警。即使微小偏差也可能嚴重扭曲結論。

SRM 是實驗可信度的第一道防線——分析結果之前，先檢查分流是不是壞了。

### 3. Simpson's Paradox

子群體各自顯示 B 版勝出，但合併數據後 A 版反而勝出。根因是混淆變數導致各組樣本比例不均。解法：確保隨機分配正確，做分層分析。

### 4. Novelty Effect 和 Primacy Effect

新功能初期因「新鮮感」表現偏高（Novelty Effect），舊用戶對介面改動不適應導致對照組有不公平優勢（Primacy Effect）。

實驗至少跑滿一個完整週期（含週末），觀察指標隨時間的收斂趨勢。對新舊用戶分開分析。

### 5. Network Effects

在社交網路或雙邊市場中，用戶間互相影響使得 SUTVA（Stable Unit Treatment Value Assumption）被違反。DoorDash 等平台用 Switchback Experiments（以時間切片而非用戶分群來切換 Treatment / Control）解決這個問題。

## 進階方法

### CUPED（Microsoft 提出）

用實驗前數據作為共變量縮減變異數。如果用戶的實驗前行為跟實驗中行為高度相關，可以藉此去除個體間固有差異。Bing 實測達到約 50% 變異數縮減——等同加倍流量或減半實驗時間。

### Multi-Armed Bandit

在實驗中動態調整流量，把更多流量導向表現較佳的變體。適合時效性高的場景，但只能針對單一指標最佳化，且無法提供嚴謹的統計推論。

我認為重大決策用 AB Test，短期促銷或多變體探索用 MAB。

### Interleaving（交錯實驗）

專為排序 / 推薦系統設計。兩個演算法的結果交錯混合呈現給同一用戶（Team-Draft 演算法），根據用戶點擊歸屬判斷偏好。靈敏度遠高於傳統 AB Test。

## 監控與決策框架

### 護欄指標（Guardrail Metrics）

<!-- 📸 建議截圖：決策流程圖——SRM → 護欄 → 主指標的四步決策判斷流程（無需脫敏） -->

實驗的安全網。Airbnb 的實踐建議：
- 事前承諾（Pre-commitment）：實驗前就定義每個護欄的可接受損害閾值
- 分類：品質指標（crash rate、latency）、商業指標（revenue、retention）、平台指標（SRM 檢查）
- 數量控制：2-3 個護欄即可，過多會增加假陽性風險

### 決策流程

1. SRM 通過嗎？不通過 → 結果不可信
2. 護欄指標安全嗎？不安全 → 即使主指標正向也需審慎
3. 主要指標達 MDE 且統計顯著？是 → 推進上線
4. 主要指標不顯著？→ 評估是否需更多流量或放棄

---

設計 AB Testing 系統最大的學習是：統計學只是一半，另一半是**工程可靠性**。分流有 bug、Bloom Filter 假陽性失控、星球生命週期沒管好——任何一個環節出問題，統計分析就在錯誤的基礎上做推論。

我還在想的一個問題是：當產品的用戶量不夠大（KOL Radar 不是千萬 DAU 的產品），很多實驗的 MDE 設定不得不放寬，導致只能偵測到比較粗的效應差異。有沒有辦法在小流量的 SaaS 產品上，更有效率地跑實驗？CUPED 是一個方向，但還需要更多實作經驗才知道在我們的場景下能縮減多少變異數。
