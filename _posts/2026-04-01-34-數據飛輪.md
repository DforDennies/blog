用戶搜尋 → 點擊 → 收藏。這些行為數據回過頭來訓練推薦模型。但這個飛輪轉起來比想像中難。

在 iKala AI Lab 做了兩年多的 TPM，經手搜尋引擎（Hermes）、相似網紅推薦（Similar KOL / CHAOS）、標籤分類（General Tag）、行為萃取（TextAction）。這些系統理論上都能形成數據飛輪——用戶行為改善模型，模型改善體驗，更多用戶帶來更多數據。實務上，飛輪的每個環節都有摩擦力。

## 相似網紅：行為信號怎麼進模型


Similar KOL（內部代號 CHAOS）從 V1 到 V3 的演進，就是一個數據飛輪逐步建立的過程。

**V1 沒有行為信號。** CHAOS 1.0 用的是純內容特徵——TF-IDF 向量、Hashtag 共現、Mention 共現。把每個 KOL 的貼文內容轉成稀疏矩陣，算 pairwise similarity。這個方法的問題是：兩個 KOL 講類似的話題，不代表對品牌方來說他們是可替換的。

**V2 引入用戶行為。** 2024 年，V2 把用戶的瀏覽行為納入模型。從 GA4 事件中提取 I2I（Item-to-Item）關係——用戶在一次 session 裡依序看了哪些 KOL。這些瀏覽序列餵進 DeepWalk 演算法，學出每個 KOL 的 32 維向量表徵。用戶的行為隱含了「這兩個 KOL 常被一起看」的信號，這比純文字相似度有意義得多。

V2 的效果確實比 V1 好，在 2024 年 12 月的會議上被確認。

但行為信號的引入也帶來新問題。DeepWalk 需要足夠的行為序列才能學到好的 embedding。日本市場有 59 萬 KOL，行為數據比台灣少很多，IG-JP 的計算跑了 17 小時，而且 Spot Node 隨時可能被回收。冷啟動更麻煩——新 KOL 沒有行為歷史，DeepWalk 生不出 embedding，只能 fallback 到 Tag-based 隨機採樣或純隨機。

**V3 的資料源遷移。** 2025 年底的 V3 把資料源從 BigQuery 搬到 Trino / Iceberg，ID 體系從 UUID 切到 DC encode_id。算法架構沒有大改，但程式碼做了大量優化——SQL 修正讓 PRD TW-IG 的資料量下降 98%，batch 處理讓 Memory 下降 46%。

## 搜尋品質飛輪：Hermes 的困境


Hermes 搜尋引擎的飛輪理論上更直覺：用戶搜尋 → 點擊結果 → 點擊數據當作 relevance signal → 改善排序。

但實際上，Hermes 的 feedback loop 一直是未解問題。

Hermes v1.0 用 facebook/mcontriever-msmarco（768 維），v1.5 換成自訓的 MiniLM（壓縮到 64 維），搜尋品質確實提升了——NDCG@10/100 都比純 BM25 好，推論從 387ms 降到 138ms。但這些改善來自模型架構和壓縮技術，不是來自用戶行為的回饋。

問題出在「如何收集使用者 feedback 改善搜尋品質」這一步。搜尋行為數據有，Amplitude 裡記錄了 Search for KOL、Visit KOL Detail 等事件。但要把這些數據轉化成 relevance label（這個搜尋結果到底好不好），需要的是 offline evaluation pipeline，而且必須處理 position bias（排在前面的結果天然被點更多）。

Search Quality 是 AI Lab 最長壽的專案（2023-06 到 2025-05），核心指標是 CTR、CSQI（搜尋品質指數）、ESI。北極星指標定義為「增加每用戶的 influencer collections」。但從指標定義到模型改善，中間的自動化程度還不夠。

## 標籤分類飛輪：General Tag 的 Feedback Loop

General Tag 有一個有趣的 feedback 機制：透過 property 回饋持續改善。

KOL Radar 的用戶在搜尋時會使用 General Tag 做篩選。如果某個標籤的搜尋結果品質差，用戶會反映。2025 年 6 月就有用戶反映相似網紅不夠貼切，排查後發現是 General Tag 的準確度問題——周杰倫被標成「運動」因為他代言運動品牌，男女朋友互稱「寶寶」被標成「親子」。

這種 feedback 是質化的，不是量化的。品牌方不會告訴你「這個標籤的 precision 是 0.85」，他們只會說「推薦的 KOL 不準」。要從這句話推導出「General Tag 的美妝類別把運動品牌代言算成運動」，中間需要大量的 debug。

更根本的問題是：General Tag 從 NeuralClassifier（HMTC）轉向 LLM 標註之後，feedback loop 的性質變了。傳統模型的 feedback 是「標錯了，重新標註一批資料，重新訓練」。LLM 的 feedback 是「改 prompt」。改 prompt 的效果立竿見影，但不保證在其他案例上不退步。

## 推薦系統飛輪：ALS 的閉環

ALS（Alternating Least Squares）是飛輪做得最完整的一個。它是協同過濾——基於 KOL 和品牌的業配歷史做推薦。247,833 筆業配紀錄、15,627 位 KOL、51,179 個業配帳號，超參數 n_factor=256，MAP@10 = 0.06，Hit@10 = 0.225。

ALS 的數據來源是 TextSponsor 模型 → NER → 品牌 fuzzy match。也就是說，飛輪的起點是另一個 AI 模型的輸出。如果 TextSponsor 把某篇非業配文誤判為業配文，這筆錯誤數據就會進入 ALS 的訓練集。

DCR（Deep Content Recommendation）嘗試解決冷啟動問題，用 NER + Autotag 特徵做 KOL embedding。但效果不理想——IG 多數網紅的文章類型不集中，embedding 學習困難。這代表內容特徵在某些場景下不足以支撐推薦品質，行為信號才是關鍵。

## 飛輪的五個摩擦點


把上面的經驗歸納：

**1. 冷啟動。** 新 KOL、新國家、新平台都沒有行為數據。Similar KOL V2 的解法是多階段 fallback（Tag-based → Random），但 fallback 的推薦品質遠低於有行為信號的結果。

**2. 數據品質。** 飛輪依賴高品質的行為數據，但行為數據天然帶噪。用戶的一次「點擊」不代表「認可」，可能只是好奇。TextSponsor 的誤判會污染 ALS 的訓練集。General Tag 的錯誤標籤會讓搜尋結果變差。

**3. Feedback 延遲。** 從用戶行為到模型改善之間的時間差。Similar KOL 是定期排程跑批次計算，不是即時更新。Hermes 的搜尋品質改善更慢，需要累積足夠的行為數據才能做一次 offline evaluation。

**4. 指標和模型的脫節。** 產品指標（CTR、收藏率、有點比）和模型指標（NDCG、F-score、MRD）之間不是一對一的關係。模型指標提升不一定帶來產品指標的改善，反過來也是。

**5. 跨系統依賴。** Similar KOL 依賴 General Tag 的品質。General Tag 依賴 TextLangID 的語言偵測。搜尋品質依賴 Hermes 的 embedding 品質。任何一個環節的退步都會沿著依賴鏈傳遞。

## 飛輪轉起來的前提

我認為數據飛輪不是自然發生的，需要刻意設計幾件事：

- **行為事件的標準化**：Similar KOL V3 從 GA4 事件切換到 AI Lab 自有的 Event Guides 系統，就是為了更好地控制事件的定義和品質。
- **離線 evaluation pipeline**：不能只看線上 A/B test，要有 offline 的品質衡量。Similar KOL 用 MRD（Mean Rank Difference），Hermes 用 NDCG@10/100，TextAction-En 用加入空標後的 accuracy。
- **品質閘門**：NLPFactory 的 100 筆人工驗收、TextAction-En 的 reward model 二次過濾、General Tag 的 LLM-as-a-Judge 月度稽核。每個環節都需要品質閘門。

## PM 在飛輪裡做什麼？

數據飛輪裡 PM 不只是寫 spec 的人，至少有三件事是工程師不會主動做的：

**決定什麼行為值得收。** 不是所有點擊都有意義。用戶在搜尋結果頁停留 0.3 秒就滑走，這個「曝光」算不算有效數據？我們在 Similar KOL V3 定義過：點擊後停留超過 5 秒才算 positive signal，純曝光不進訓練集。這個閾值不是 ML 工程師決定的，是 PM 看了用戶行為分佈後拍板的。

**設計讓 feedback 跑回來的 UI。** Explicit feedback（收藏、加入商案）比 implicit feedback（點擊、瀏覽）乾淨很多，但用戶懶得給。所以 UI 設計要在不打擾的前提下多收一點信號——收藏按鈕放在卡片上而不是詳情頁裡，「不感興趣」的選項用手勢滑掉而不是點按鈕。這些小設計直接影響飛輪的數據品質。

**決定 model retrain 的觸發時機。** ML 工程師傾向定期 retrain（每週、每月），但 PM 要看的是產品指標——搜尋點擊率連續兩週下降、推薦的收藏轉化率低於閾值，這時候才是真正需要 retrain 的時機。不是技術排程驅動，是業務信號驅動。

---

飛輪的每個環節都需要人盯。行為數據品質會退化，模型效果會漂移，用戶使用模式會改變。飛輪不是建好就不管的東西。

還有一個我沒想清楚的問題：當 Similar KOL 的推薦越來越準，用戶是不是反而會看更少的 KOL？準確的推薦可能縮小探索範圍，而探索範圍的縮小又會減少行為數據的多樣性。飛輪有沒有可能轉到某個點之後開始減速？
