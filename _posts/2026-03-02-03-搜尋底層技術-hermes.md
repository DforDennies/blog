搜尋引擎要支援七種語言，每種斷詞邏輯完全不同。

韓文有黏著語的語尾變化，日文有三套書寫系統要正規化，英文要做 stemming 把 running 還原成 run，中文要靠統計模型判斷哪裡該斷。KOL Radar 同時服務台灣、日本、韓國、印尼、越南、泰國、馬來西亞，搜尋引擎的 analyzer 設計不是一個「通用方案」能搞定的事。

Hermes 是 KOL Radar 的核心搜尋引擎，採用 BM25 + Embedding 的混合搜尋架構。這篇記錄它的技術設計，以及我們怎麼把 384 維的 embedding 壓到 64 維還讓搜尋品質變更好。

## 為什麼混合搜尋

BM25 和 Embedding 搜尋各有盲點。

BM25 靠的是詞頻和逆文件頻率，擅長精確匹配。使用者搜「雅詩蘭黛」，BM25 一定能找到提到「雅詩蘭黛」的貼文。但如果 KOL 的貼文只寫了「Estée Lauder」沒寫中文，BM25 就找不到了。

Embedding 搜尋靠的是語意相似度，把文字轉成向量，在向量空間裡找最接近的。「雅詩蘭黛」和「Estée Lauder」的向量會很接近，語意搜尋能找到。但 embedding 有時候會過度泛化 — 你搜一個小眾品牌名，它可能把整個美妝類別的結果都丟給你。

Hermes 的做法是兩者結合：先用 BM25 做初步檢索，再用 embedding 做 reranking。這樣既保留精確匹配的能力，又補上語意理解的維度。

## 七語系 Analyzer 設計

Elasticsearch 的 analyzer 決定了文字怎麼被拆解和索引。Hermes 為不同語言配了不同的 analyzer：

**韓文（ko_analyzer）**：使用 Korean tokenizer。韓文的語尾變化複雜，「먹었습니다」（吃了）和「먹다」（吃）需要被還原到同一個詞根。

**日文（ja_kuromoji_analyzer）**：用 kuromoji 分詞器，加上 baseform filter 做原形還原、part_of_speech filter 過濾助詞、cjk_width filter 處理全半形轉換、ja_stop 停用詞、stemmer 做詞幹提取。日文的處理鏈最長，因為它混用漢字、平假名、片假名三套系統，加上外來語（カタカナ）大量存在。

**英文（en_analyzer）**：standard tokenizer + lowercase + english_stop + english_stemmer，再加上 indonesian filters。英文和印尼文共用一個 analyzer 是因為印尼文大量借用英文詞彙，底層處理邏輯相似。

中文、越南文、泰文、馬來文各有對應的處理方式，但核心思路一致：tokenize → normalize → filter → stem/lemmatize。

Query 策略用的是 multi_match 的 best_fields 模式，tie_breaker 設 0.5。這意味著搜尋時會在原始文字欄位和各語言子欄位上同時匹配，最高分的欄位作為主分數，其他欄位的分數乘以 0.5 加入。這種設計讓跨語言搜尋自然發生 — 一個搜尋可以同時命中中文內容和英文內容。

## Embedding 壓縮：384 維降到 64 維

這是整個專案最核心的研究問題。Hermes 的 embedding 搜尋用的是 paraphrase-multilingual-MiniLM-L12-v2，原始 384 維。每筆 KOL profile 都需要存一個 384 維的向量在 ES index 裡，當 KOL 數量到達數十萬等級，index 的儲存空間和搜尋延遲都成為問題。

目標：把維度降下來，但搜尋品質不能掉。

### 模型選型

<!-- 📸 建議截圖：MiniLM vs mcontriever 模型比較表——維度、多語言支援、benchmark 結果對照（無需脫敏，為公開模型數據） -->

比較了兩個 baseline：

- paraphrase-multilingual-MiniLM-L12-v2（384 維）
- mcontriever（768 維）

理論上 mcontriever 的 768 維應該比 MiniLM 的 384 維有更好的表達力。但實測發現，人工抽樣評估和 benchmark 結果不一致 — mcontriever 並沒有達到預期的 2 倍優勢。最終選擇 MiniLM 作為基礎，因為它在多語言場景的表現夠穩定，而且維度更低，壓縮空間更大。

### 降維實驗

試了三種方法：

1. **直接 PCA 降維**：把現有 embedding 投影到低維空間，看不同維度保留多少資訊
2. **Query Generation 重訓**：參考 Spotify 的 "Improving Retrievability in Search with Query Generation" 和 UDAPDR 方法，用 LLM 生成 query 來增強小模型的訓練數據
3. **Triplet Training**：用 (query, positive_passage, negative_passage) 三元組訓練。Hard negative 的取得方式很關鍵 — 用 ES 的 BM25 檢索取前 5 個結果作為 hard negative，這些是字面上相關但語意上不匹配的結果，最能教模型分辨細微差異

訓練時測了 batch size 16、32、256 多種配置，epoch 1 和 7 都試過。一個有趣的發現：cosine similarity 和 dot product 的差異很小，真正影響品質的是 output dimension。

### 最終結論

<!-- 📸 建議截圖：Embedding 降維效果對比圖——384 維 vs 64 維在各平台驗證資料上的搜尋品質與回應時間（無需脫敏，為效能數據） -->

全平台資料訓練並降維至 64 維的 MiniLM，在各平台的驗證資料表現上都遠優於其他配置，也優於只用 BM25 的搜尋方式。搜尋回應時間也有明顯加速。

從 384 維降到 64 維，儲存空間直接降到原來的六分之一，向量計算的延遲也大幅下降。

## 訓練數據與評估

訓練數據來自社群平台的真實搜尋場景，主要是 Instagram 貼文，涵蓋寵物、美妝、食品等品類，查詢類型包括品牌搜尋（雅詩蘭黛、寵物食品等），覆蓋 zh、en、ja、ko、id 五種語言。

<!-- 📸 建議截圖：Rolling Forward 資料切分示意圖——Train/Valid/Test 的時間區間和向前滾動方式（無需脫敏，為方法論圖） -->

資料切分用的是時間序列的 rolling forward 策略：訓練用 2020-01 到 2021-10 的資料，驗證用訓練結束後 3 個月，測試再後 3 個月。每個 fold 向前滾動 1 個月。

第一個 fold 效果特別差，因為它結束在 2021-05，剛好是資料最缺失的時段。這件事給我的教訓是：時間序列的資料切分如果不考慮資料密度，某些 fold 的結果根本不能信。

評估方面，我們最初用人工抽樣，但發現人工評估和 hidden dim benchmark 不一致（mcontriever 理論上應該大幅優於 MiniLM，但人工評估沒反映出來）。最終改用自動化 benchmark，結果更穩定也更可復現。

## 部署架構

最終的部署配置：

- Encoder：MiniLM 降維至 64 維，Matryoshka 或 PCA
- 訓練：In-batch negatives + BM25 hard negatives
- ES 查詢：multi_match + best_fields + tie_breaker=0.5
- 多語言：各語言獨立 analyzer，共用一個 embedding 模型

共用 embedding 模型是一個重要的設計決策。如果每個語言一個 embedding 模型，維護成本和部署複雜度都會爆炸。MiniLM 本身是多語言預訓練的，加上 triplet training 用的是多語言資料，一個模型就能處理所有語言的語意搜尋。

---

BM25 + 64 維 embedding 的組合跑了一段時間，效果和效率之間找到了還算滿意的平衡。但我一直在想：query 端和 document 端現在共用一個 embedding 模型，如果改成 asymmetric architecture——query 用輕量 encoder，document 用更重的 encoder 預先算好——能不能在不增加線上延遲的情況下再推一步？KOL profile 的內容更新頻率不高，document 端的 embedding 可以離線慢慢算，這個場景天然適合 asymmetric 的設計。但要加一個 encoder 就多一份維護成本，值不值得還不確定。
