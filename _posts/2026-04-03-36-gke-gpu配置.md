一張 A100 一個月 $2,000。但你可能只需要 T4 甚至 CPU。

5 台 GPU 機器、96 個任務、從 RTX 3090 到 A6000 到 RTX 6000 Ada，每張卡的使用率差異很大。這篇是 GPU / CPU 規格選擇和 Spot vs On-demand 策略的實戰經驗。

## 五台機器，五種用途

先看現狀。iKala AI Lab 的 GPU 基礎設施是 IDC 自建 + GCP 混合架構。

**ai-server-01（原 IDC-321）：NVIDIA RTX A6000 x 8。** 這是 SRE 建置的 GPU Platform，過去跑所有 NLP 模型的訓練：textadvertiser、textsentiment、textsegment、textlangid、textcleaning、socialprofiler、hermes、generaltag、textemotion、textkeyphrase、textknowledge、textaction、CV Tagger。光列名字就有 13 個模型。

**IDC-322：NVIDIA RTX 6000 Ada x 8。** 專門做三件事：傳統 ML 模型的重訓（每月 LLM-as-a-Judge 不合格就重訓或增量訓練）、LLM-as-a-Judge 稽核（Kolr Palette / SaaS 品質檢查）、Model Performance Detection（月度線上模型稽核）。

**IDC-323：NVIDIA RTX 3090 x 2。** 已經不做訓練了，轉型為 Hybrid Cloud 建構用。generalTag（zh / en / jp）和 textaction（zh / en / jp）的推論已經遷移到這裡。

**ikala 2：NVIDIA RTX 3090 x 2。** 中小型任務。測試開源小模型（gpt-oss-20b、Qwen2.5-VL-7b）、外部使用（台大實習生、BE 團隊 script）、研華 PDF extract / embedding / OCR。

**ikala 4：NVIDIA RTX A6000 x 8。** 主力中的主力。96 個任務裡有 67 個指定用 A6000 x8。用途最多元：LLM 貼標（常駐）、相似網紅 / 潛力網紅的大矩陣運算、開源大模型（gpt-oss-120b、Qwen2.5-VL-32B）、國泰健康專案。

## 任務分佈看資源缺口

96 個任務的資源需求分佈很不均勻：

- A6000 x 8：67 個任務
- A6000 x 4：12 個任務
- RTX 3090 x 2：6 個任務
- RTX 6000 Ada x 5：1 個任務

67/96 的任務需要 A6000 x8，但我們只有兩台（ai-server-01 和 ikala 4）。瓶頸很明顯。

優先級方面，High 有 41 個，Medium 有 49 個，Low 只有 6 個。已完成 30 個，進行中 5 個，未開始 61 個。大量的任務還在排隊。

## Labeling 是最大宗用途

看完任務分佈之後我的觀察是：GPU 的最大消耗者不是模型訓練，是 LLM 貼標。

General Tag 的 LLM 標註排程排到了 2029 年。每個模型 x 每個平台 x 每個市場都需要獨立的 labeling 流程：

**TW 市場**：General Tag 的 FB 已完成（2026-01），IG 和 Threads 進行中，YouTube 排在 2026-03~05。後面還有 Text Advertiser、Text Action、NER、Social Profiler、Emotion、Textkeyphrase、Ontology——一路排到 2029 年。

**JP 市場**：General Tag 的 IG 排在 2029 年之後。

每個 labeling 任務跑在 ikala 4 或 IDC-322 上，用的是 gpt-oss 120B + Qwen3-VL 8B。台灣 FB 的 180M posts 標了 171M，這不是幾小時能跑完的事。

## Spot vs On-demand 的抉擇

<!-- 📸 建議截圖：Spot vs On-demand 決策矩陣——批次計算用 Spot + checkpoint / 線上服務用 On-demand 的選擇邏輯（無需脫敏） -->

在 GKE 上跑批次計算時，Spot 實例的成本大約是 On-demand 的三分之一。但 Spot 隨時可能被回收。

Similar KOL V2 的 IG-JP 計算需要 17 小時。跑在 GKE 的 Spot Node（n2-highmem-8）上。問題來了：17 小時的計算，中間 Spot Node 被回收怎麼辦？

解法是 checkpoint + retry。find_top 佔整體計算時間的 75%，所以把 find_top 分 batch 處理，每個 batch 完成後 checkpoint 到 GCS，被回收之後從最近的 checkpoint 接續。

Dev / Staging 環境盡量用 Spot。Production 的批次計算（像 Similar KOL）也用 Spot，但要有完整的 checkpoint 機制。Production 的線上服務（像 Hermes 的即時推論）必須用 On-demand，不能有中斷風險。

## CPU vs GPU 的選擇

不是所有模型都需要 GPU。iKala AI Lab 的部署模式有四種：

**ONNX Runtime（CPU）**：Hermes encoder（MiniLM 64 維，model.run 4.9ms）、Autoprofile CV（EfficientNet V2-S）、general_tag（18M / 27M 模型，1000 筆約 30 秒）。這些模型轉完 ONNX 之後，CPU 推論的速度已經夠用了。

**PyTorch（GPU）**：textaction 和 textkeyphrase 用的是 mt5-base（462M），生成式模型需要 GPU。batch_size 只能開到 1，開到 8 就 OOM。

**FastText（CPU）**：autotag v1.5、textsponsor、textlangid。FastText 完全不需要 GPU，連 ONNX 都不用，直接 CPU 推論。

**Elasticsearch native（無額外計算）**：BM25 + dense_vector kNN 直接在 ES 裡做，Hermes 的混合檢索不需要外部 GPU 服務。

我認為最被低估的選項是 ONNX Runtime on CPU。很多人預設 ML 模型就需要 GPU，但 18M 的 general_tag 模型在 CPU 上跑得很快。把 GPU 留給真正需要的任務（大模型推論、LLM 標註、大矩陣計算），CPU 處理輕量推論綽綽有餘。

## IDC vs Cloud 的混合策略

2026 年初有一個重要的架構決策：捨棄 ML Server 的運行方式，改為 OCI（Open Container Initiative）交付。

過去的模式：ML 工程師在 IDC 機器上手動部署模型，跑在自己管理的 ML Server 上。問題是維運成本高，擴展不方便。

新的模式：RD 把模型打包成 OCI image，上傳到 Google Artifact Registry（GAR）。SRE 自由選擇用 docker 還是 k8s 部署。predict model 統一走 OCI 上 GAR，開源 model 初期手動部署，中長期自動化。

這個轉變讓 IDC 的機器可以更靈活地分配。ai-server-01 的 A6000 x8 過去被 NLP 模型訓練佔滿，現在模型打包上 GAR 之後，訓練任務可以用完就釋放，不需要長期佔用。

IDC-323 的 Hybrid Cloud 建構也是這個方向。把 generalTag 和 textaction 的推論遷移到 IDC-323 的 RTX 3090 x2 上，減少 GKE 的計算成本。

## 成本控制的幾個具體數字

<!-- 📸 建議截圖：成本優化項目清單——ES disk / ONNX 轉換 / MiniLM 壓縮 / BQ partition 的節省金額匯總表（保留美元數字） -->

- ES hot tier disk 從 pd-ssd 換成 pd-balanced：省 $84 / 月（300Gi x 4 nodes）
- ES warm-cold disk 同樣的操作：省 $49 / 月
- ONNX Runtime 取代 PyTorch 推論：TextAction-En 從 15.4 秒 / 1000 筆降到 8.1 秒，相當於計算資源需求砍半
- MiniLM 64 維 vs mcontriever 768 維：推論時間 4.9ms vs 9.7ms，embedding 存儲空間差 12 倍
- BigQuery 加 partition 和時間 range 之後：避免了一次 $10,569 的全表 scan

這些數字單獨看不大，但乘以 15+ 個 cluster、數十個服務、12 個月，累積效果很可觀。

---

核心問題不是「要買哪張卡」，是「什麼任務該用 GPU、什麼不該用」。96 個任務裡，真正需要 A6000 x8 的可能不到一半。但因為歷史原因和排程便利性，大家習慣性地往最大的機器丟。

我一直在想的是：LLM 標註的 workload 排到 2029 年，但 2029 年的 GPU 效能和成本一定跟今天不一樣。現在為 2029 年的 labeling 排 GPU 時程，是不是某種程度上的過早最佳化？
