ML 團隊 North Star 是增加每位用戶的網紅收藏數量。搜尋品質怎麼量化？

KolRadar 的使用路徑：搜尋 → 瀏覽結果 → 進入 KOL Detail 頁 → 收藏或解鎖。搜尋是第一步，搜尋品質差，後面的漏斗全部掉。「搜尋品質」是一個很模糊的概念——到底好不好？好在哪裡？比上個月好還是差？需要一個可量化的指標。

## 三大指標

DS 團隊從 2023 年開始設計搜尋品質指標，最終定義了三個互補的指標：

**CTR（Click-Through Rate）**——最直接的指標。搜尋之後有沒有點擊結果。這裡有個 SaaS 搜尋的特殊性：CTR 可以超過 100%，因為一次搜尋可以點很多個結果。所以另外定義了 ECTR（Effective CTR）：只看「至少點了一次」的搜尋佔比，上限 100%。

**CSQI（Comprehensive Search Quality Index）**——綜合搜尋品質指數，0-100 分。這是主指標，底下拆成多個子指標加權彙整。

**ESI（Engagement Search Index）**——搜尋互動品質指數。2024 年 7 月正式化。

CSQI 是核心，架構長這樣：

```
CSQI (0-100)
├── 搜尋結果滿意度
│   ├── 互動加權評分 (IWS) — 參考 DCG
│   ├── 有效搜尋率 (ESR)
│   ├── 點搜比 (CTR)
│   ├── 未點擊率
│   └── 搜尋停留時間
├── 內容契合度
│   ├── KD 頁停留時間
│   └── 解鎖回訪率
└── [未來] 使用體驗 + 整體滿意度 (NPS)
```

## 互動加權評分：DCG 的變體

CSQI 裡最有意思的子指標是 IWS（Interaction Weighted Score），概念來自資訊檢索的 DCG（Discounted Cumulative Gain）。

傳統 DCG 衡量的是「排名靠前的結果是不是比較相關」。IWS 做了一個改動：不用人工標記的相關性分數，而是用用戶的實際行為來推斷：

| 分數 | 行為 |
|------|------|
| 1 | 進入 KOL Detail 頁 |
| 2 | Stay KOL Detail ≥ 1 次（停留約 10 秒） |
| 3 | Stay KOL Detail ≥ 3 次（有往下滑） |
| 4 | Stay KOL Detail ≥ 5 次，或點了 Similar KOL |
| 5 | 收藏、解鎖、IRM、Deep Report 等關鍵動作 |

折損因子跟 DCG 一樣：`discount = log₂(rank + 1)`。排名第一的結果被點擊，價值最高；排名靠後的結果被點擊，折損後的價值較低。

SQL 實作：

```sql
SELECT kolId, interaction_score,
  LOG(new_rank + 1) / LOG(2) AS discount_factor,
  interaction_score / (LOG(new_rank + 1) / LOG(2)) AS discounted_score
FROM click_processed
```

### 計算單位：Session

一個重要的設計決策：CSQI 以 Session 為計算單位，不是以單次搜尋。原因是用戶的搜尋滿意度是整個 Session 的累積感受——可能搜了三次才找到想要的網紅，但整體還是滿意的。

Session 的統計特徵：中位數 3 次搜尋（20,054 sessions 統計），P90 持續 53 分鐘。超過 600 分鐘的 Session 視為異常排除。

計算對象只看訂閱用戶，排除 ikala 內部帳號和異常 workspace。

## IWS 月度趨勢

2024 年的 IWS 月度數據：

| 月份 | IWS |
|------|-----|
| 01 月 | 1.18 |
| 02 月 | 0.98 |
| 03 月 | 1.02 |
| 04 月 | 1.05 |
| 05 月 | 0.93 |
| 06 月 | 0.41 |
| 07 月 | 0.27 |
| 08 月 | 0.77 |

6-7 月明顯下降。這跟點搜比的分析結果吻合——5 月串接 DC 推薦演算法之後，搜尋品質全面下降。IWS 在 8 月開始回升，也跟後續的 DC 改版時間對得上。

不過 IWS 的精確度受埋點品質影響很大。`Stay KOL Detail` 事件的觸發率直接決定了分數 2-4 級的判定，如果埋點不穩定，分數就會波動。

## 搜尋技術棧

搜尋品質指標的改善需要搜尋系統本身的配合。KolRadar 的搜尋技術棧在過去兩年經歷了幾輪演進：

<!-- 📸 建議截圖：搜尋技術棧演進時間線——2023 Hermes v1.0 → 2024 排序優化 → 2025 多模態的三階段（無需脫敏） -->

**2023：Hermes v1.0**——BM25 + Embedding 混合搜尋。Elasticsearch 做 BM25 後端，多語言句子嵌入（中/日/英），RRF（Reciprocal Rank Fusion）融合兩路結果。

**2024：排序優化年**——
- Action Search 上線（行為維度搜尋）
- 分詞規則 V1/V2。V2 的關鍵改善是**完全匹配優先**——如果 query 完全命中某個網紅名字，直接排第一，不走 tokenized 匹配
- 雙引號精確搜尋修復（跨語言大小寫）
- 關鍵字命中片段——用戶可以看到搜尋結果裡哪些字被匹配到

**2025：多模態**——image-to-image 搜尋 POC、Related Keyword Agent（LLM 擴展搜尋詞）、Similar KOL V3（行為+嵌入混合推薦）。

### 排序架構

搜尋排序分兩步：

1. **Candidate Expansion（召回擴展）**——NQE（Neural Query Expansion）和 NFE（Neural Field Expansion）。目標是「找到更多相關的候選人」，探索導向
2. **Candidate Ranking（排序）**——Reranker model，目標是 CTR 提升，利用導向

這是經典的 exploration-exploitation 架構。目前有一個痛點：搜尋永遠不會回傳空結果（即使 query 完全不相關也會有結果），這讓用戶很難判斷「搜尋有沒有用」。

## 已知搜尋問題

1. **搜尋邏輯使用率極低**——AND 只有 3.25%、"-"（排除）0.92%、OR 0.87%。搜尋指引沒有存在感
2. **長 query 失敗**——超過 8 個字的 query 容易搜不到東西
3. **30 日留存 ~50%**——搜尋品質直接影響留存

## 可復用框架

CSQI 的設計邏輯可以套用到任何「搜尋 → 瀏覽 → 深入互動」的 SaaS 產品：

1. 定義 Session 邊界（時間上限、關鍵事件）
2. 設計互動性分數（根據產品核心行為分 3-5 級）
3. 參考 DCG 做排名加權（前面結果被點擊價值更高）
4. 多子指標加權彙整為單一分數
5. 月度追蹤 + 子指標拆解定位問題

<!-- 📸 建議截圖：CSQI 可復用框架的五步驟圖——適用於不同 SaaS 產品的通用化設計（無需脫敏） -->

重點是第 2 步。互動性分數的定義完全取決於產品——電商可能是「加入購物車 > 點擊 > 瀏覽」，求職平台可能是「投遞 > 儲存 > 點進詳情」。核心思路一樣：用行為推斷滿意度，越「深」的行為代表越高的滿意。

---

我認為 CSQI 最大的價值不是那個分數本身，而是它提供了一個「搜尋品質下降了，是哪裡出問題？」的診斷框架。6-7 月 IWS 驟降的時候，可以明確指向「用戶在搜尋結果上的互動深度變淺了」，進而追溯到 DC 串接的改動。如果只有一個黑盒分數，下降了也不知道該改什麼。

還在想的一個問題：IWS 目前用的是 DCG 的 log 折損，假設排名越前面越重要。但如果用戶的搜尋習慣是「先快速掃過前幾個，然後跳到第二頁找特定類型的 KOL」，那 log 折損可能低估了後排結果的價值。要怎麼從行為數據裡反推「用戶真正的掃描模式」，這件事我還沒有答案。
