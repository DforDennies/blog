每個 ML 模型自己包一個 Flask API，部署時要記住 10 個不同的 port。這是 KolRadar AI Lab 過去的模型服務方式 — 能用，但不 scale。

團隊維護 30+ 個 ML 模型，從 NLP 到 CV 到推薦系統，每個部署方式不一致、監控困難、擴展性差。統一化不是可選的工程任務，是存活問題。

## 舊架構的痛點


舊架構的問題很直接：

```
每個模型自訂 API endpoint
├── 部署不一致（有的用 Flask，有的用 FastAPI，有的直接 gRPC）
├── 監控困難（每個模型要各自設定 metrics）
├── 擴展性差（scaling 邏輯各寫各的）
└── 模型更新要動到服務程式碼
```

每次 ML Engineer 訓練完新模型，部署就是一場冒險。不同的框架、不同的 container 設定、不同的 health check 方式。出了問題排查也困難，因為每個服務的 logging 格式都不一樣。

遷移路徑經歷了三個階段，而且最終方向跟原本預期的不太一樣。

**Phase 1-2（2025 H1）**：初始模型遷移到 MLServer，部署流程標準化。MLServer 的目標是做統一推論服務 — 一個框架支援所有模型的 serving。

**Phase 3（2025 Q4）**：textadvertiser 和 generaltag 遷移到 MLServer 完成。目標是 50%+ predict models 遷移到 MLServer。

**轉向（2026-01）**：放棄 MLServer，改用 OCI 容器。

為什麼放棄？MLServer 作為統一框架有它的限制 — 特別是當你要同時服務傳統 ML 模型和 LLM/VLM 時，一個框架很難兼顧兩者的需求。改用 OCI 容器後，每個模型打包成獨立的 Docker image，LLM 和 VLM 分到不同 image，上傳到 GAR（Google Artifact Registry），部署到 IDC 或雲端。

新的流程更靈活：

```
1. 開源模型打包 → Docker image
2. LLM/VLM 模型分開到不同 image
3. 上傳到 GAR
4. 部署到 IDC 或雲端
```

General Tag 的部署是個好範例：開源模型打包成 Docker image → 上傳 GAR → Dynamic lvl1/lvl2 tags from DB → 分離 LLM 和 VLM 到不同 image。

## 硬體升級：從 3090 到 6000 Ada


GPU 資源也跟著升級。IDC-321 的 VM 重構從 5+2+1 變成 7+1 配置，GPU 從 RTX 3090 x 2 升到 RTX 6000 Ada x 8。

目前部署的開源模型包含 gpt-oss-120b（Qwen/LLaMA family）和 Qwen3-VL-8B（視覺語言模型），需要 8 張 RTX 6000 Ada。

硬體清單：
- IDC-321：8x A6000，主要推論
- IDC-322：8x GPU，標註用
- ikala4（辦公室）：8x GPU，開發/測試
- ada：RTX 6000 Ada x 8，新增推論

2025-12 開始每個任務追蹤 GPU hour，做精確成本分配。已知問題包含 ikala4 GPU constraints 和 IDC NFS 100G 資料遺失事件。

## 80:20 雲端:地端目標

<div class="mermaid">
pie title 推論工作量分配
    "雲端 API (商業模型)" : 80
    "地端 IDC (開源模型)" : 20
</div>

成本控制的核心策略是 80:20 — 80% 推論在雲端（API calls），20% 在地端（IDC）。

```
├── 80% 推論在雲端（API calls，用商業模型）
├── 20% 推論在地端（IDC，用開源模型）
├── predict models 逐步遷移到 OCI + GAR
├── 開源 LLM 評估 → 降低 API 成本
└── Taiwan LLM Leaderboard 輔助決策
```

這個比例不是隨便定的。商業 LLM（GPT-4、Claude）的品質確實比自建模型好，但成本也高。開源 LLM 部署在 IDC 可以降低 API 成本，但 GPU 資源有限、維護成本也是真的。

一個已知的不對稱問題：production model quality < LLM quality。生產模型精度有限，LLM 推論品質更好但成本高。蒸餾策略是縮小差距的方向 — 用 LLM 的輸出來訓練更小的模型。

## 配套工具鏈

新架構搭配了幾個工具：

- **MLflow**：實驗追蹤與模型版本管理
- **Superset**：效能監控儀表板
- **GAR（Google Artifact Registry）**：Docker image 倉庫
- **GPU Hour Tracking**：每任務 GPU 小時追蹤（2025-12 正式化）

## StarRocks 遷移

附帶提一下，模型服務的 feature store 也在變。PostgreSQL 磁碟使用量過大，需要存儲轉移。評估過 GCS content extraction（放棄）、PG compression（放棄），最後選定 StarRocks。

要求 zero downtime migration、schema splitting、PG size calculation（無法 live resize）。理想 Q2 2026 完成，實際可能延到 Q3。

---

從 MLServer 到 OCI 容器看起來像走了彎路。但需求從「統一傳統 ML serving」擴展到「同時服務 ML + LLM + VLM」，原來的方案確實不夠用。核心不是選哪個框架，是建立一套所有人都能遵守的部署規範。

我還在想的是：80:20 的雲端和地端比例到底怎麼動態調整？當開源模型品質持續追上商業模型，這個比例應該會往地端傾斜。但 GPU 的採購和維護需要提前規劃，不是說調就能調的。什麼時候該加碼投資 IDC 的 GPU，什麼時候該繼續吃商業 API 的紅利，這個判斷目前還是靠直覺多過數據。
