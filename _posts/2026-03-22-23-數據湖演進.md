BigQuery 每月 $1,200 vs Trino/Iceberg $466。這不是理論試算，是我們實際在 KolRadar 跑出來的數字。

近 500 萬 KOL、每季 746 GB Post 資料、六個平台的即時數據。離線計算全壓在 BigQuery on-demand 定價上，單月費用飆到 $2,407 的時候，就知道該認真考慮替代方案了。

## 五個階段的演進


整個數據架構經歷了五個階段，每次演進都有很明確的觸發原因：

**Phase 1**（~2023）：PostgreSQL + Elasticsearch + BigQuery。PG 做 OLTP，ES 做搜尋和 Data Lake（post 按季度分 index），BQ 做離線計算和報表。這是最原始的架構，什麼都能用，但成本不可控。

**Phase 2**（2023-11）：BQ 費用爆量觸發 P0 事件。PRD 月費達 $2,407，單一 SQL 最大成本約 400 GB（~$3.2）。開始設 Daily Quota 限制、監控不符規定的 SQL、程式設定 `maximum_bytes_billed = 100GB` 做保護。

**Phase 3**（2024-04 ~ 2024-07）：Trino POC + Iceberg 引入。目標取代 BQ 作為 Data Lakehouse，擺脫 BQ 束縛。

**Phase 4**（2024-10 ~ 2025）：Trino 全面上線，ES → Iceberg 和 PG → Iceberg Pipeline 建立。

**Phase 5**（2026-02 起）：StarRocks 遷移規劃。因為 Trino 的 JOIN 效能瓶頸和不支援 Upsert，決定再往下走一步。

## Trino 架構怎麼搭


Trino 的核心價值是統一查詢 — 一個 SQL 引擎可以同時查 PG、ES、BQ、Iceberg 四種資料來源。

PRD 的 Coordinator 開 2 CPU / 16 GB（理想是 4 CPU / 32 GB），Worker 開 8 CPU / 32 GB。關鍵原則是 Coordinator 不兼任 Worker 角色。

Auto-scaling 策略基於 Query count 和 CPU usage。等待中的查詢數量（SubmittedQueries - CompletedQueries - RunningQueries）除以 6 取整決定需要多少 worker。CPU 超過 80% 也會觸發擴容。常駐 1 個 worker pod，等待狀態持續 3 分鐘則加 1 pod。

四個 Connector 各有特性：

- **Iceberg Connector**：用 Nessie 做 catalog（Git-like 版控），Parquet + Snappy 格式，GCS 做儲存。Filesystem Cache 基於 Alluxio，啟用後讀取時間減少 99%
- **ES Connector**：用 Scroll API，每個 worker 可達 30 MB/s。PRD 單 ES node `max_scroll_context = 500`，要分配給 Specific 爬蟲（180）、Platform 爬蟲（60）、ES 2 Iceberg（36）、Data Mart Backfill（224）
- **BQ Connector**：直接設定 Catalog 就能取用 BQ Tables
- **PG Connector**：直接取用 PG Tables，用於小表 JOIN

Resource Groups 分兩大類：Cron Jobs Warehouse（OLAP high-volume queries）和 Data Center（high concurrency small queries）。平均保留資源，單一情境運行時可佔用全叢集。

## Iceberg Table 設計


Namespace 按來源分類：`data_lake_elasticsearch`（ES 搬過來的）、`data_lake_postgresql_data`（PG data schema）、`data_lake_postgresql_ai_portal`（PG ai_portal schema）、`data_temp`（ETL 暫存）。

分區策略用 `_index`（ES index 名，遷移追蹤用）和 `month(post_time_epoch)`（時間分區過濾）。檔案格式 Parquet + Snappy，目標檔案大小 1 GB。

一個重要的限制：Iceberg 不支援 JSON 寫入，需要用 `JSON_FORMAT()` 轉成 String。

維護機制由 Argo CronJob 每天執行：合併小檔案（optimize，目標 128MB 以上）、標記過期 snapshot（1 天 retention）、清除過期的實體檔案。Nessie 不會自動清理 dropped tables 的 GCS files，需要額外處理。Schema 下的 table 有上限 500 個，也需要監控。

## 效能實測

幾個關鍵的測試結果：

**半年 IG Post**：460 GB / 6,000 萬筆 + JOIN 210 萬 Profile。ES → GCS Iceberg 花 1.5-2 小時，GCS Iceberg 之間的讀+寫+JOIN 30 分鐘內完成。

**一口氣算全部 IG 帳號統計**：四張大表全由 ES 讀取，小表由 PG 讀取，最複雜的 SQL 查詢順利完成。

**ES → Iceberg 效能瓶頸**：直接 Merge 完整執行要 2.03 小時，分月份 merge 反而更慢（2.24 小時）。先 CTAS 到暫存表再 merge 只要 52.55 分鐘 — 這是推薦的方式。

**Filesystem Cache**：啟用後第二次以後的查詢幾乎 100% 從 cache 讀取，時間減少 99%。但因為無法針對特定 table cache，TTL 設 1 小時做折衷。

## 成本對比


直接看數字：

- BigQuery on-demand：月費 $800 ~ $1,200（正常），峰值 $2,407
- Trino 低配（c2d-highcpu-4 spot）：空轉 $46.62 / 滿載 $466.2
- Trino 高配（c2d-highcpu-8 spot）：空轉 $83.23 / 滿載 $832.3
- 固定成本：Coordinator + Nessie 約 $29/月
- GCS 儲存：~$24.28/TiB/月

低配滿載 $466 vs BQ 平均 $1,000，成本確實砍了 50% 以上。加上 Auto-scaling，大部分時間不需要滿載運行，實際節省更多。

## 為什麼又要換 StarRocks

Trino 有幾個根本性的限制：

1. Coordinator 是單點故障，沒有原生 HA
2. Java-based，SIMD 利用受限
3. 不支援 Upsert
4. JOIN 效能受限於 MPP Java 框架

StarRocks 是 C++ 寫的，原生 SIMD + 完全向量化，MPP + In-memory shuffle，支援 Upsert（Delete-Insert pattern），FE/BE 分離可做 HA。Benchmark 結果：Iceberg 查詢速度比 Trino 快 5.54 倍。

遷移計畫 2026-02 啟動，Phase 1 涵蓋 PG Migration（Logical Replication）、ETL 順序調整（Crawler → StarRocks）、Cron Job 改寫（100% 拋棄 BQ）。Looker 維持 BQ 來源，走 StarRocks → External GCS Iceberg → BQ → Looker 的路徑。

共約 60+ cron jobs 需要處理，其中約 30 個要改寫到 StarRocks，約 20 個純 BQ 可以 SKIP，約 5 個 Trino/Iceberg 相關的可以 Sunset。開發 + 移轉期間總投資約 $8,142。

---

從 BQ 到 Trino/Iceberg 是一次成功的降本，但 Trino 的 JOIN 效能瓶頸又推著我們往 StarRocks 走。每次遷移都有收益，也都有代價。

我還沒想清楚的是：每次換引擎都有遷移成本，60+ cron jobs 的改寫不是小事。什麼時候該忍受現有方案的不完美，什麼時候該下定決心遷移？這個判斷標準到現在還沒有一個讓我完全滿意的框架。
