每天 80,000 個帳號要爬、600 萬帳號要維護。這是 KolRadar 爬蟲系統的日常，不是峰值。

產品覆蓋 IG、FB、YouTube、TikTok、Threads 六大平台，每個平台 API 規則不同、反爬機制不同、資料結構不同。寫個 script 爬一個站很簡單，但要同時搞定六個，需要的是微服務架構、狀態機驅動、Proxy 輪換的分散式系統。

先說結論：沒有一個統一的方法可以搞定所有平台。每個平台都是一場獨立的戰爭。

IG 用 `instagram.com/api/graphql` 逆向端點，需要帶 `x-ig-app-id`、`x-csrftoken`、`x-fb-lsd` 三個 header，doc_id 指定 query 類型。Reels 有 List、Single、bulk-route-definitions 三種 API，各自不同的呼叫方式。

FB 更混亂。Post 跟 Reels 的抓取邏輯不一致，FB 內部 ID mapping 混亂到你得從 HTML meta tag 逆向取 `link_profile_id`，因為 `platform_user_id` 跟 `link_profile_id` 是兩個不同的東西。

TikTok 有 Research API（官方，需審核），限制 600 req/min，每次最多 100 筆 Video、前 1000 則 Comment。逆向部分需要處理 browser fingerprint、challenge、token 驗證，還要解析 `SIGI_STATE` script tag。

Threads 是 2025 年加入策略爬蟲的新平台，帳號總數 1,288,202。沒有 cookie 只能抓 25 篇貼文，有 cookie 才能抓全部。2025-05 Meta 開始強制 csrftoken header，直接觸發了一個 P1 事件 — 所有 Threads 爬蟲一夜之間 403。

YouTube 表面上有官方 API，但 quota 限制嚴格。逆向部分需要解析多個 JS script 檔案，參數散佈在不同地方，才能拼出完整的請求。

我認為這就是為什麼爬蟲系統不能外包、不能用現成方案的原因 — 每個平台的 API 生態都是活的，一年調整 2-3 次結構，內部欄位調整甚至每月都有。

## 微服務怎麼拆

爬蟲系統拆成三個核心服務：

1. **Crawler Micro**：所有平台的 Fetch/Save gRPC API，統一爬取介面。Go 寫的，用 go-micro 框架
2. **Crawler Consumer**：從 Redis Queue 消費任務，驅動 FSM（有限狀態機）進行爬取流程
3. **Crawler Background**：定時找完成的 task，通知 Datahub API 觸發 ETL


一個完整的爬取流程長這樣：

```
App → Datahub API: create crawler task
Datahub API → Crawler Micro: create task (gRPC)
Crawler Micro → Redis: push to profile queue
Crawler Consumer: pop task → fetch profile → push to post queue
... (post → comment 依序處理)
Crawler Background: find complete task → notify Datahub API
Datahub API → ETL: trigger pipeline
```

服務間通訊主要靠 gRPC，任務佇列用 Redis Queue（profile queue → post queue → comment queue 三階段），事件分發用 Kafka（7 天 retention，基礎 500 GB），任務狀態持久化在 PostgreSQL。

FSM 狀態機驅動整個流程：`init → crawling → crawled → etl_done`。這個設計的好處是任務可以在任何階段中斷後自動恢復，壞處是有些 edge case 不好處理 — Threads comment 就曾經出現中斷不自動恢復的 bug。

Service Registry 從 Consul 遷移到 Kubernetes Registry（go-micro plugins），避免 Consul 節點頻繁換的維護成本。部署在 K8s 上，DEV / STG / PRD 三環境，Proxy 三環境共用。Feature Toggle 用 `UseMicroService` 控制新舊 Crawler Micro 切換。

## 排程策略：不是每個帳號都一樣重要

600 萬帳號不可能每天都爬。DS 團隊根據粉絲數產出 follower base rules，決定每個帳號的爬取頻率。


以 IG 為例，2,568,483 個帳號的分布：
- high_frequency：29.84%（766K 個帳號）
- medium_frequency：63.50%（1.6M）
- low_frequency：6.66%（171K）

每日爬取帳號數平均 81,152，最大 85,940，最小 70,635。

策略爬蟲的排程由 Argo Workflow 管理，基於 follower base rule 匯入 `crawler_state` table 後自動排程。常態爬蟲由外部 API 觸發，指定爬蟲則是手動觸發特定帳號。

Country Isolation 是另一個排程維度。主力國家（台灣、日本）優先一個月更新，非主力三個月。後續新增新加坡和印尼時，策略演進方向是從「每天啟動」改為「每 N 小時啟動」，用時區分 slice，工作日和週末分配不同任務量。

## 反爬蟲對抗


Proxy Pool 由 HAProxy 管理。每個電信帳號可撥 6 個 IP，節點每 20 分鐘重新撥接取得新 IP。Slack channel `crawler-proxy-traffic` 定時流量回報，`crawler-proxy-tunnel` 回報 IP 更新。

各平台的 Rate Limiting 應對策略不同：

- TikTok Research API 600 req/min，按規定使用
- IG 逆向依 cookie 狀態變動，輪換 cookie 並監控 error rate
- Threads 用 Cookie Loader/Taker 機制管理 cookie pool
- FB 定期 refresh URL 避免 link_profile_id 變動

監控指標包含爬蟲完成速率（accounts/hour）、Throughput（requests/min）、Error rate（error requests/all requests），分類錯誤類型追蹤。

## 數據最終去哪


爬完的資料先存 GCS（爬蟲完成的定義 = 資料放到 GCS），然後走 ETL pipeline：GCS2Flat → Flat2Latest。清洗轉換後分別寫入 Elasticsearch（KOL 搜尋和 Post 搜尋）、BigQuery（長期分析）、Kafka（事件暫存和 EventHub 分發）。

ES 上 PRD 的 post 資料估計每季約 746 GB（平均每 KOL 15 post/季，每 post 11kb）。Redis Sentinel 負責策略爬蟲 80,000 entries，常態爬蟲額外需要 200,000 entries。

每萬帳號的爬取成本約 0.975 USD，只計算爬取完成放到 GCS 的階段。

---

爬蟲系統最難的不是爬取本身，而是維護。每個平台都在不斷調整 API、加強反爬、改變資料結構。你今天跑得好好的程式，明天可能因為一個 header 的變動就全部 403。

我還在想的問題是：當平台數量持續增加（PTT、Dcard、News、RSS 都在擴充中），微服務的粒度要拆到多細才是對的？目前 Crawler Micro 是一個統一的 gRPC 服務處理所有平台，但每個平台的邏輯差異越來越大，是不是該按平台拆成獨立服務？
