95% 準確率聽起來很好。但對於 19 萬 KOL，5% 錯誤就是 9,500 個標錯的結果。

30 多個 ML 模型在線上跑。標籤分類、行為萃取、業配偵測、語言偵測、推薦排序——每個都有自己的準確率數字。我學到的是：AI 產品的不確定性管理不是技術問題，是系統設計問題。模型會錯，這是確定的。問題是錯了之後會發生什麼。

## 62 起生產事故的教訓


iKala 的 Platform 團隊記錄了 62 起生產事故。其中 P0（最高嚴重度）佔了 48%，將近一半。我把跟 AI / ML 相關的事故拉出來看，發現一個 pattern：模型本身出錯的事故很少，但模型的不確定性引發的「系統性問題」很多。

**DB / SQL 問題是最大痛點。** 62 起事故裡有 12 起（19%）是 DB / SQL 相關。典型情境：SQL 查詢未優化、Index 缺失、MySQL optimizer 選錯 index。這些不是模型問題，但 AI 系統的高吞吐量（每天 80K 爬蟲任務、180M 筆 General Tag 標註）把 DB 的弱點放大了。

**OOM 是第二大殺手。** 7 起 OOM 事故。最典型的是 CDP campaign worker 把整份名單讀進記憶體一次送出，沒有分批。還有 Radar 13.2.6 版本，gorm scan 一次把 2.63GB 的資料載到 heap。AI 服務處理的資料量天然就大，如果不做分批，OOM 只是時間問題。

**部署引發的事故有 6 起。** 付款功能失效（migration AUTO 先跑但服務還沒更新）、Stripe Webhook 跨版本遺漏 tag、MySQL config 改了但服務沒重啟。AI 模型的上線頻率比一般服務高（每個模型都有自己的版本週期），部署風險也跟著放大。

## 模型錯誤的傳遞鏈


AI 系統最危險的不是單一模型出錯，是錯誤沿著依賴鏈傳遞。

在 iKala 的架構裡：

```
TextLangID (語言偵測) → 所有多語言服務
TextSponsor (業配偵測) → ALS 推薦系統的訓練資料
General Tag (標籤分類) → Similar KOL 的特徵輸入
NER → Brand Database fuzzy match → ALS/DCR
```

如果 TextLangID 把一篇日文貼文判成中文，後面所有中文模型都會拿到一筆垃圾輸入。TextLangID 的準確率 94.89%，看起來不錯，但換個角度算：平台上如果有 1000 萬筆貼文，就有 51 萬筆的語言被判錯。這些判錯的貼文會進入下游的 Autotag、Keyphrase、TextAction 處理，產出一批品質不確定的結果。

General Tag 的誤判更直接影響用戶。2025 年 6 月，用戶反映相似網紅不準。我們排查下來，根因是 General Tag 的語意誤判——運動品牌代言被標成「運動」而不是「美妝」。Similar KOL 用 General Tag 的 Top 3 標籤做特徵，標籤錯了，推薦自然不準。

## 八大根因模式

從 62 起事故歸納出八大根因模式，我認為其中五個跟 AI 系統的不確定性管理直接相關：

**模式 1：SQL 查詢未優化。** 6+ 起。AI 服務的批次查詢量大（例如 TextAction 要對整個 country 的貼文做標註），如果 WHERE 條件的型態錯了（int vs string），MySQL 會選錯 index，CPU 直接 100%。

**模式 2：記憶體管理不當。** 5+ 起。AI 模型的推論通常需要載入大量資料（KOL 列表、貼文內容、embedding 向量），如果 ORM 一次性載入過多資料到 heap，GC 無法及時清理就 OOM。

**模式 4：外部服務依賴變更。** 4+ 起。Instagram 逆向 API 無預警失效是最典型的。爬蟲系統每天 80K 的 IG 任務依賴這些 API，一旦失效，下游的所有 NLP pipeline 就斷料。Alert 的延遲偵測是 1 小時，太慢了。

**模式 6：成本失控。** 3+ 起。BigQuery 的一次全表 scan（沒加時間範圍）花了 $10,569。Cloud Logging 因為 Istio 配置錯誤噴 log，一個月 $1,477。AI 服務的計算量大，成本失控的幅度也大。

**模式 7：重試風暴。** 3+ 起。Kolr 全站無回應的那次，根因是 DB 記憶體滿 → App retry → DB 更重 → 重啟 → App 繼續 retry 的循環。AI 服務的 retry 策略如果沒有 backoff 和 circuit breaker，就是雪崩觸發器。

## Postmortem 裡反覆出現的教訓

看完 25 份事故報告的 postmortem，有幾條教訓反覆出現：

**大量資料操作必須分批。** CDP campaign worker 的 OOM 事故寫得很明白：「會把整份名單讀進 mem，再一次送出，沒有分批次寄送」。AI 服務處理的資料量從幾千筆到幾百萬筆，分批是基本功。

**Staging 必須模擬 Production 規模。** 12.21.0 在 staging 跑得好好的，上 prod 就爆。因為 staging 和 prod 的資料量差了一個數量級。AI 模型特別容易遇到這個問題——在小資料集上看不出來的效能瓶頸，到了 production 的規模就爆開。

**CI/CD 中 Migration 必須有人為卡控。** AI 模型的上版常伴隨 schema 變更（新增欄位、修改 index mapping）。如果 migration AUTO 先跑，但對應的服務還沒更新，就會出問題。

**Domain Knowledge 不能集中在一兩人。** AI Lab 有一次事故，兩位核心工程師同時請假，問題處理速度大幅下降。每個 AI 模型都有自己的 domain knowledge，如果只有一個人懂，就是單點風險。

## 改善：從告警到閘門

2024 年是事故高峰年（36 起），跟 KOL Radar v12-v13 大版本重構和多個架構遷移高度吻合。2025 年降到 8 起，2026 年到目前只有 2 起。下降的原因是一系列系統性的改進：

- Cloud Armor 規則阻擋爬蟲和異常流量
- MySQL 記憶體升級 15GB → 24GB + swap alerting
- CI/CD pipeline 加人為卡控 migration
- BigQuery 強制 partition + 時間 range
- 新服務上線必做壓測（100 / 1K request）
- 低風險時段 infra 變更 policy

針對 AI 模型的不確定性，我認為最有效的改善是「品質閘門」：

- **NLPFactory**：每個任務 100 筆人工驗收
- **TextAction-En**：Reward model 二次過濾
- **General Tag**：LLM-as-a-Judge 月度稽核（2025-04 建立）
- **Similar KOL**：A/B Testing 框架（Anubis 切流），穩定後每週更新報告
- **Model Performance Detection**：MLflow + Superset 指標監控（2025-01 建立）

## 不確定性的三種管理策略


把上面的經驗歸納成三種策略：

**接受不確定性，設計 fallback。** Similar KOL 的五階段推薦策略就是這個思路。Stage 1 用最精確的方法（Content-Based + CF），如果結果不夠就 fallback 到 Stage 2（Tag-Based），再不夠 Stage 3（Pure CF），最後 Stage 4（Random）。每一層的品質遞減，但保證一定有輸出。

**降低不確定性，加品質閘門。** TextAction-En 加入空標（+25% accuracy）和 reward model 過濾（+10% accuracy）。NLPFactory 的採樣策略確保 topic diversity。這些都是在資料層面降低不確定性。

**隔離不確定性，避免傳遞。** TextKnowledge 先用 TextSell 做前置過濾，只有確定是銷售文的貼文才進入完整的知識萃取 pipeline。Country Isolation 把不同國家的資料隔離到不同 ES index，避免一個國家的異常影響其他國家。

---

62 起事故、48% 是 P0、月均 3 起降到 0.7 起。數字在改善，但問題沒有被「解決」。模型永遠會犯錯，外部 API 永遠會變更，資料品質永遠會退化。能做的是讓錯誤影響範圍可控、恢復速度夠快。

但有一件事我一直沒想通：品質閘門越多，上線速度越慢。100 筆人工驗收、reward model 過濾、A/B test、月度稽核——每個閘門都是時間成本。在「快速上線」和「品質保證」之間，最佳的平衡點在哪裡？這可能沒有通用答案，取決於錯誤的代價有多高。
