兩個 B2B 客製化專案，一個成功延續，一個不續約。

Client B（製造業）的工業文件解析 API，F1 score 做到 95%，單文件成本 $0.03 美元，順利結案。Client C（健康管理）的 LINE 聊天機器人，acceptance score 72.4 分（門檻 70），通過驗收，但客戶決定不繼續。

這兩個案子是 iKala AI Lab 首次重要的外部企業 API 交付，也是 40925 Agent 架構的第一次實戰測試。

## 為什麼這兩個案子重要

AI Lab 過去的產出都是內部服務——TextSponsor 偵測業配、Similar KOL 推薦演算法、General Tag 分類系統。這些模型跑在 KOL Radar 平台裡面，用戶感覺不到 AI Lab 的存在。

Client B和Client C是第一次直接面對外部客戶，自己定規格、自己交付、自己負責品質。這代表幾件事：
- 需要建立外部 API 交付的 SOP
- 需要定義清楚的評估標準（不是「看起來 OK」就行）
- 需要估算成本基準（為後續客戶報價提供參考）

## Client B：工業文件解析

### 專案背景


Client B（製造業）需要把工業技術手冊（PDF 格式）自動解析成結構化資料。手冊裡有表格、圖片、規格參數，格式不統一。

專案從 2024 年 4 月啟動，到 2025 年 Q1 結案，前後約 12 個月。

### Phase 1：傳統 NLP

第一階段用開源的 MinerU 做 PDF 解析。API 分成四個子項目（7.1 到 7.4），各自有不同的萃取目標。

F1 score 的結果是 95% 和 71%——兩個子任務差距很大。95% 的那個子任務是結構化程度高的欄位萃取，71% 的那個涉及更多非結構化的段落理解。

2024 年 5 月完成 BERT score 評估，同時確認推論機器規格。到 2024 年 9 月 Phase 1 穩定化。

### Phase 2：VLM 整合

第二階段引入 GPT-4o 處理複雜表格和圖片解析——傳統 NLP 搞不定的部分。

成本是關鍵數字：**單文件約 $0.03 美元**。這個數字後來成為其他客戶報價的基準線。F1 average 超過 90%。

2025 年初做了 prompt optimization，query topic 分類的 accuracy 達到 78%。

### 商業價值

Client B案驗證了三件事：
1. AI Lab 有能力做 B2B 服務交付
2. 傳統 NLP + VLM 的混合架構在工業文件場景是可行的
3. 成本可控——$0.03 / 文件的數字讓後續報價有底

## Client C（健康管理）：LINE 聊天機器人

### 專案背景

Client C（健康管理）需要一個 LINE-based 的健康管理聊天機器人。用戶在 LINE 上問健康相關問題，機器人自動回答。

專案從 2024 年 11 月 kick-off，到 2025 年 12 月確認不續約，前後約 13 個月。

### 技術選型

這個案子直接用了 40925 Agent 的基礎設施作為底層。推論模型是 GPT 4.1。

Phase 2 的 acceptance testing 結果：**score 72.4 / 目標 70**。通過了，但只通過了一點點。Phase 3 acceptance 也完成了。

2025 年 7 月完成所有驗收，但到 12 月 22 日客戶確認不繼續。

### 結案原因

不續約是客戶端的決策，**不是技術問題**。驗收分數通過了、功能交付了、SLA 也滿足了。

我認為這是 B2B AI 專案常見的狀況：技術達標不代表商業價值被認可。客戶內部可能認為 ROI 不足、組織優先級調整、或是預算被砍。這些不是我們能控制的。

## 兩個案子的比較

| 維度 | Client B | Client C |
|------|------|------|
| 類型 | 文件解析 API | LINE 聊天機器人 |
| 技術底層 | MinerU + GPT-4o VLM | 40925 Agent |
| 評估指標 | F1 95%/71%, ACC 78% | Score 72.4/70 |
| 單位成本 | ~$0.03 / 文件 | — |
| 結果 | 成功交付 | 不續約 |
| 持續時間 | ~12 個月 | ~13 個月 |
| 用途 | 驗證 B2B API 交付能力 | 驗證 Agent 架構 |

### 共用基礎設施

兩個案子都使用了 NLPFactory + 40925 Agent 架構。這代表 AI Lab 的基礎設施可以被不同場景復用——工業文件解析和健康聊天機器人看起來完全不同，但底層的 LLM 調用、prompt 管理、評估框架是共享的。

## 學到的事

### 評估標準要提前定義


Client B用 F1 score，Client C用 acceptance score。兩種評估方式各有優缺：F1 是自動化的、可重複的，但需要標註數據；acceptance score 是人工打分的、主觀的，但更貼近客戶感受。

我認為未來的企業交付應該**同時用兩種**：自動化指標確保模型品質沒有退化，人工評分確保用戶體驗達標。

### 成本估算比想像中重要

Client B案的 $0.03 / 文件是一個里程碑數字。有了這個基準，後續客戶報價就不用靠猜。

成本的組成包括：LLM API 調用費、推論機器費用、開發人力。其中 LLM API 調用費是最好估的（按 token 計費），推論機器費用取決於是用自建 GPU 還是雲端 API，開發人力是最難估的。

### 「通過驗收」不等於「客戶滿意」

Client C案最大的教訓。72.4 分通過了 70 分的門檻，但通過門檻和讓客戶願意續約是兩件事。


如果重來，我會在專案中期就開始做 business review——不只追蹤技術指標，也追蹤客戶內部的使用率、stakeholder 滿意度、續約意向。等到驗收完才發現客戶不想續，已經太晚了。

---

兩個案子走完，AI Lab 從「內部服務提供者」跨出了第一步。SOP 建起來了、成本基準有了、技術架構也驗證了。

但我還在想一個問題：AI Lab 做 B2B 客製專案的商業模式到底對不對？每個客戶都需要大量的客製化（Client B要解析工業手冊、Client C要做健康問答），這跟 SaaS 的「一次開發、多次銷售」邏輯完全相反。除非能找到一種方式把客製化的部分標準化——比如把 40925 Agent 框架做成可配置的平台，讓不同客戶只需要調整 prompt 和資料源。這個方向值不值得追，我還沒有定論。
