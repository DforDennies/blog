有一次客戶問業務：「你們的資料多久更新一次？」業務轉頭問我，我轉頭問 DE。DE 回了一張架構圖，上面畫了八個方塊，每個方塊裡面還有子系統。我看了十秒，決定自己重畫一張，重新理解業務場景與功能。不是每一層我都寫過 code，但每一層我都跟對應的工程師討論過「為什麼選這個技術」。答案往往不是「因為它最好」，而是「因為在當時的條件下，它最合理」。

## 先看全景：一筆資料走過的八個關卡

一位 Instagram KOL 發了一則貼文。從這則貼文被我們系統抓到，到品牌方在 KOL Radar 搜尋時看到它，中間經過這些：

八層，分屬不同團隊維護。爬蟲是 AI Lab Crawler 團隊的活，ML 加工是 AI Lab 的活，數據服務層是 Backend Team 的活。我的角色是把這些串起來，確保每一層的輸出是下一層預期的輸入。

聽起來很理所當然，但實務上「串起來」這件事比想像中難。光是「爬蟲完成」的定義，不同人理解就不同：爬蟲工程師認為資料放到 GCS 就是完成，DE 認為 ETL 跑完才算，PM 認為用戶能搜到才算。

## ① 爬蟲層：每天 80,000 個帳號，六個平台

爬蟲系統覆蓋 IG、FB、YouTube、TikTok、Threads、PTT，加上新聞和 Dcard。每天大約爬 80,000 個平台帳號，KOL 總量將近 475 萬。

核心是用 Go 寫的 Crawler Micro，所有平台的 Fetch/Save 都走 gRPC 統一介面。任務透過 Redis Queue 分發，Consumer 用 FSM（有限狀態機）驅動：先爬 profile，再爬 post，最後爬 comment，按序執行。

**為什麼選 Go？** 爬蟲是 I/O 密集型任務，goroutine 的併發模型比 Python 的 GIL 友好太多。公司後端標準語言也是 Go，工具庫（`kernel_kit_go`）共用。

**為什麼用 Redis Queue 而不是 RabbitMQ？** 問了 DE 才知道，FSM 狀態管理需要原子操作，Redis 天然支援。爬蟲任務的生命週期很短（profile → post → comment），不需要 RabbitMQ 那套複雜的 routing 和 acknowledgment。對這個場景來說，Redis 就是最輕量的選擇。

爬取頻率不是固定的。DS 團隊根據粉絲數算出每個帳號的爬取頻率——高粉絲的帳號爬得勤（IG 約 30% 歸類為 high frequency），低粉絲的三個月才爬一次。這套規則寫在 PG 的 `follower_base_rules` 表裡，策略爬蟲每天自動從這張表排程。

反爬蟲對抗是永恆的戰爭。Meta 2025 年 5 月突然開始強制 csrftoken header，Threads 爬蟲整批 403，觸發了 P1 事件。TikTok 的 browser fingerprint、IG 的 GraphQL 逆向、FB 的 link_profile_id 謎題——每個平台都有自己的坑。Proxy 層用 HAProxy 做 IP 輪換，每 20 分鐘換一批 IP。

## ② 原始儲存 + ③ ETL：GCS → PG flat → PG latest

爬蟲完成的定義 = 資料放到 GCS。這是一個很明確的邊界：爬蟲團隊只要確保 raw JSON 安全落地到 GCS，後面的清洗不歸他們管。

ETL 分兩步：`GCS2Flat`（解析 JSON 寫入 PG flat table）和 `Flat2Latest`（更新 latest table）。中間做資料去重、欄位標準化、時間格式統一。整個 ETL 用 Argo Workflow 編排，K8s 原生的 DAG，失敗自動重試。

**為什麼分兩步而不是一步到位？** 跟 DE 討論後理解到，flat table 是回溯用的。如果 latest 出了問題，可以從 flat 重跑，不用回去重爬。而且 latest table 的更新會觸發 CDC——這是下一層的入口。

## ④ CDC 即時同步：PG 改了一筆，幾秒內到 ES

這是我覺得整條管線裡最精巧的一段。PostgreSQL 的 latest table 被 ETL 更新後，Debezium 從 WAL（Write-Ahead Log）捕捉變更，送進 Kafka，再由 Flink 寫入 Elasticsearch。

**為什麼選 Debezium 而不是 Google DataStream？** 當時 DataStream 對 PostgreSQL 的支援只在孵化階段。Debezium 是成熟方案，而且零侵入——它讀 PG 的 WAL，不改任何應用程式碼。相比之下，如果在應用層做雙寫（同時寫 PG 和 ES），任何一個 bug 都可能導致兩邊資料不一致。

**為什麼選 Kafka 而不是用 PubSub 就好？** 其實兩個都在用，但定位不同。Kafka 負責高吞吐持久化串流（CDC 管線、EventHub），7 天 retention 可以重播；PubSub 負責事件驅動場景（CDP 事件、爬蟲觸發），GCP 託管免維護。Kafka 由 DE Team 管，PubSub 由 Backend Team 管，分工清楚。

Kafka PRD 環境 1.8TB，高流量 Topic（IG Post）設 18 個 partition，壓縮用 lz4。跟 DE 確認過的吞吐量：每秒約 4,000 rows，consumer lag 穩定在 0。

**為什麼 Flink 而不是 Spark Streaming？** Flink 是真正的事件驅動串流，不是微批次。對 CDC 場景來說，延遲差距很明顯。而且 Flink 有 Savepoint/Checkpoint 機制，掛了能精確恢復。Flink 從 PyFlink 演進到 Java Flink，中間還經歷了 ScyllaDB 作為中間層被退役的過程——最終確認直接寫 ES 就夠了，多一層 ScyllaDB 只增加複雜度。

Flink 同時寫三個 Sink：ES 負責即時搜尋，BigQuery 負責長期分析、DS query 和月度排程 ETL 的資料源（K-Score、audience、follower history 等衍生特徵都從 BQ 拉），Iceberg 是正在測試的下一代 Data Lakehouse，目標取代部分 BQ 的角色並把成本從月費 $2,400 砍到 $466。

有個真實踩到的坑值得一提：Debezium 的 TOAST Values 問題。PG 的 `pgoutput` plugin 在 UPDATE 時，如果 TOAST 欄位沒被改，WAL 裡就不會包含這個欄位的值——Debezium 收到的是 `__debezium_unavailable_value`。解法是 `ALTER TABLE REPLICA IDENTITY FULL`，但會增加 CPU 負擔。這種問題沒踩過根本不會想到。

## ⑤ ML 加工：15 個模型，把原始資料變成可搜尋的特徵

資料進了 ES 還不夠。一則 Instagram 貼文此時只有原始欄位：文字、圖片、讚數、留言數。品牌方搜尋時需要的是：「這則貼文是什麼主題？」「是不是業配？」「KOL 幾歲？」「情感正面還是負面？」

這些全靠 ML 模型加工。結果寫回 PG 的 `ml_*` tables，再透過 CDC 同步回 ES。

15+ 個模型按功能分三大類：

**文本分析**——判斷貼文「在說什麼」和「怎麼說」：
- `textlangid`：語言偵測。用 FastLangID，選它是因為社群短文本上推理速度遠比 Google NLP API 快，成本也低。
- `textsponsor`：業配文偵測。用 FastText，不需要 GPU 就能跑多語言。
- `textaction`：行為萃取，分成教學、推薦、開箱等六大類。用 mt5-base seq2seq，T5 架構天然適合分類 + 生成，而且 mt5 的多語言預訓練覆蓋 7 個語系。
- `Sentiment`：情感分析。BERT fine-tune，在分類任務上就是穩。

**標籤系統**——給貼文和 KOL 打標籤：
- `General Tag`：通用主題標籤（L1 + L2 階層），用 sentence-transformers 做 embedding 再分類。
- `Autotag v2.0`：27 類貼文主題，從 FastText 演進到深度學習。選 FastText 起步是因為快且便宜，跨語言和 multi-label 搞不定後才升級。

**KOL Profile**——補齊 KOL 的人口統計屬性：
- `Autoprofile CV`：從頭像估計年齡和性別。用 EfficientNet V2-S 轉 ONNX，在準確率和推理速度之間取得最佳平衡。ONNX 的好處是部署不綁定框架。
- `YOLOv5x`：圖片物件偵測 599 類，給圖片內容打標。

**推薦**——計算 KOL 之間的關聯：
- `Similar KOL (CHAOS)`：用 DeepWalk 從用戶行為序列做圖嵌入，加上 General Tag 過濾，FAISS 做向量搜索。DeepWalk 比 V1 的純 TF-IDF 內容相似度效果好很多，因為它捕捉的是「用戶一起看哪些 KOL」的行為關係。

整體策略是**輕量模型優先**。FastText、MiniLM 這種小模型能解決的就不上大模型，GPU 留給真正需要的任務。所有模型盡量轉 ONNX 統一推理 runtime。而且 Hermes 自訓的 MiniLM 64 維 embedding，在社群媒體 domain 上效果居然比 OpenAI 的 text-embedding-3-small 還好——domain-specific 的小模型贏通用大模型，這件事在我們這邊被反覆驗證。

### 不只 ML 模型：26 個衍生特徵的排程計算

ML 模型之外，DataCenter 還有一整套 ETL 排程在定期計算衍生特徵。這些特徵不是即時的，而是按固定週期跑批次。跟 DE 要了一份完整清單後才發現，光是「用戶看到的 KOL 卡片上的數字」就涉及 26 種不同的計算任務，各有各的更新頻率：

**跟著爬蟲更新的（Crawler 觸發）**——爬蟲抓到新資料就重算：
- `Statistic data`：互動率、觀看率等核心指標（IG/FB/YT/TikTok/Threads，m1/m3/m6 三個時間窗口）
- `KOL URL Status`：帳號爬取狀態
- `KOL Email`：聯絡信箱
- `sponsor`：業配文標記（textsponsor 的結果落地）
- `sentiment analyze`：留言情感分析
- `word frequency`：留言關鍵字詞頻
- `hashtag statistic`：Hashtag 使用統計

**每月 15 號跑的（Monthly ETL）**——需要累積一段時間才有意義：
- `scoring`（K-Score）：KOL 綜合評分，條件是近 200 天有發文且 follower > 0
- `kol property`：從 General Tag 聚合出 KOL 的內容屬性類型，用手肘法找閾值
- `audience gender / age`：受眾性別年齡分布，需要 iqdata 或 25 筆以上有效樣本
- `lang analyzes`：語言分布，取近 12 個月有 model_output 的貼文
- `Related KOL`：品牌與網紅的互相提及（IG mention，m3/m6）
- `Estimated Reach`：預估觸及數
- `potential`：成長潛力指標，基於過去六週粉絲數、發文頻率、按讚數預測第八週粉絲

**每週跑的（Weekly ETL）**：
- `Top 5 CustomizedTag`：每個帳號最常出現的 5 個分類標籤
- `kol geo`：KOL 所在國家和城市
- `Evaluate performance（PR）`：同級距 KOL 的 benchmark 表現

這 26 個特徵有不同的 Level（KOL 級 / Platform 級 / Content 級 / Comment 級）和不同的時間窗口（none / m1 / m3 / m6）。品牌方在 KOL 卡片上看到的每一個數字背後，都對應到某個排程任務。

有些計算邏輯很講究。比如 `kol property` 的判斷規則：某個 CustomizedTag 要「連續未發文月份數不超過 N」且「近期連續發文 M 個月」且「佔整體發文數 X%」，N/M/X 每次都用手肘法自動找閾值。又比如 `scoring` 的 K-Score，如果某帳號上次能被算出 score 是 2023-10-14，latest 就會一直提供那次的結果直到下一次計算覆蓋。

這一層在架構圖上通常被省略——它不像 ML 模型那麼「性感」，但如果排程沒跑或跑錯，用戶看到的數字就是過時的。

## ⑥ ES 索引層：三套 Index + 七語系 + Country Isolation

Elasticsearch 在這個系統裡扮演了一個相當吃重的角色——它同時承擔全文搜尋、向量搜尋和聚合分析。能用一個引擎做三件事，是選 ES 而不是拆成 Pinecone + OLAP 的核心原因。

三套 Index 系統：

| 系統 | 用途 |
|------|------|
| `akashic-kol` (tw + global) | KOL 搜尋 |
| `akashic-post-{yyyy}-q{n}` (tw + global) | 貼文搜尋，按季度分 index |
| Hermes index | 語義搜尋（文找文 / 文找 KOL） |

Post 按季度分 index，是為了控制 shard 大小（官方建議 10-50GB/shard）。按月太多小 index 浪費資源，按年單 index 太大。

ILM 策略：hot → warm（180 天，forcemerge 到 1 segment）→ cold（365 天）→ delete（3 年）。這代表三年前的貼文會被刪除——品牌方真正在意的是近期表現，三年前的資料佔空間但幾乎沒有查詢需求。

**Country Isolation** 是一個我覺得設計很漂亮的架構決策。台灣是主力市場，全球 KOL 全擠在一個 index 裡查詢越來越慢。拆分方案是按國家分 index（`akashic-kol-tw` + `akashic-kol-global`），寫入時根據 `country_code` 路由，查詢時透過 ES alias 同時搜兩個 index——對上層完全透明。零停機遷移，用 Reindex 搬資料、alias 切換，Plan B 是退版 + alias 還原。

多語系搜尋支援 7 種語言：中文用 ik（分詞）、日文用 kuromoji、韓文用 nori、泰文和印尼文各有 tokenizer。每種語言的斷詞邏輯完全不同，ik_max_word（中文寫入用較細的分詞）和 ik_smart（搜尋用較粗的分詞）的搭配，是中文搜尋品質的關鍵。

**Hermes 語義搜尋**是另一層。排序分數 = BM25 分數 × cosine similarity（query embedding 和 doc embedding）。Embedding 用自訓的 MiniLM，壓縮到 64 維，推理速度 138ms/request（比原本 768 維的 387ms 快了 64%），NDCG 仍然優於純 BM25。壓縮到 64 維是儲存成本和搜尋品質的 trade-off——在社群短文本上，64 維已經足夠。

## ⑦ 數據服務層：GraphQL 統一入口

到這一步，資料在 ES 裡了。但前端不會直接打 ES——index 分散（KOL、Post 按季度分、Text Action 各自獨立），ES 查詢語法複雜。需要一個翻譯層。

**Akashic** 是 ES 的唯一封裝層，所有 ES 讀寫都經過它。這個設計的戰略價值在於：未來替換 ES 只需改 Akashic，不用動上層任何程式碼。

**為什麼選 GraphQL 而不是 REST？** 前端需要按需取欄位。KOL Search 的回傳欄位很多（基本資料、統計、標籤、相似網紅、貼文），用 REST 不是 over-fetching 就是要開一堆 endpoint。GraphQL 讓前端自己決定要什麼欄位，server 端用 Field Resolver 按需載入——如果 response 沒用到 similar_kol，就根本不查。

**為什麼用 gqlgen？** Go 生態最佳的 GraphQL 框架。Code-first、type-safe、效能好。Schema 按功能拆分成 `kol.graphqls`、`post.graphqls`、`analyze.graphqls`，維護起來清楚。

N+1 問題用 DataLoader 解。KOL Search 回 N 個結果，原本每個 KOL 各自查 similar_kol 和 related_kol。先改成 Field Resolver（按需載入），再引入 `vikstrous/dataloadgen`（gqlgen 官方推薦，評分 91/100）做批次化。初版用的 `graph-gophers/dataloader` 在高負載時記憶體會爆炸，POC 後遷移。

Pagination 用 Cursor-based，支援 ES 的 search_after，避免 deep pagination 問題。

## ⑧ 用戶端：五種搜尋方式

品牌方在 KOL Radar 上有五種找網紅的方式：

1. **Quick Search**：打名字或 ID，全文搜尋
2. **Radar Search**：篩選器（粉絲數、國家、平台、標籤）
3. **Recommend Search**：高斯方法 scoring
4. **ChatGPT Search**：LLM 自動選篩選欄位
5. **文找文（Hermes）**：語義搜尋，描述你要什麼類型的 KOL

每一種搜尋看到的每一個欄位——粉絲數、互動率、標籤、業配標記、相似網紅——都是前面七層加工的產物。用戶搜尋「台灣美妝 YouTuber」，這五個字觸發的是：ES 多語系分詞 → Hermes 語義匹配 → General Tag 過濾 → Akashic aggregation → GraphQL Field Resolver → 前端渲染。

前端透過 Gaia SDK（Go GraphQL client）跟 DataCenter 溝通。SDK 的存在讓前端不用直接碰 ES，也不用知道底下的 index 怎麼分、country isolation 怎麼路由。

## 拉遠看：為什麼是這些技術

跑完八層之後，我覺得有幾個貫穿全系統的設計哲學值得記錄。

**Go 作為核心語言**。爬蟲（I/O 密集）、API（高併發）、微服務（gRPC）——Go 的 goroutine 和編譯效能在這三個場景都是甜蜜點。整條管線除了 ML 用 Python、Flink 用 Java 之外，全是 Go。

**gRPC 作為內部通訊**。微服務之間全走 protobuf。binary 序列化比 JSON 快 5-10 倍，proto 定義即文件，breaking change 在編譯期就被抓到。protobuf 版本統一在 `kernel_kit_protobuf` repo 管理。

**CDC 而非雙寫**。選 Debezium 讀 WAL，不在應用層同時寫 PG 和 ES。零侵入、一致性有保障、7 天 Kafka retention 可以重播。應用層雙寫最怕的是部分失敗——PG 寫成功但 ES 寫失敗，或者反過來。CDC 從根本上避免了這個問題。

**ES 一引擎三功能**。全文搜尋（BM25）+ 向量搜尋（dense_vector KNN）+ 聚合分析（terms/stats/date_histogram），不用額外維護向量資料庫和 OLAP。代價是 ES 的運維確實不輕鬆（shard 管理、ILM、disk 水位），但一套系統的運維成本還是比三套低。

**K8s 原生優先**。Strimzi（Kafka）、Flink on K8s、Argo Workflow、ArgoCD——所有元件都在 K8s 上。統一資源管理、HPA 自動擴縮、Prometheus + Grafana 統一監控。IaC 全在 `datahub_sys_iac` repo 管理。

**輕量 ML 模型優先**。FastText、MiniLM、EfficientNet 這些小模型能解決的就不上大模型。自訓的 domain-specific 小模型在社群媒體場景上反覆贏過通用大模型。ONNX 統一推理 runtime，不綁定框架。

## 跑完一圈的體感

回到最初那個問題：「你們的資料多久更新一次？」

技術上的答案是：爬蟲抓到之後，經過 ETL + CDC + ML 加工 + ES indexing，到用戶能搜到，大概是分鐘到小時級別。但這個答案背後是八層架構、六個以上的團隊、幾百個 config、幾十個 Kafka topic、15 個 ML 模型在持續運轉。

我在這個過程中學到最深刻的一件事是：**系統的複雜度不在任何單一元件，而在元件之間的銜接**。Kafka 很穩、Flink 很快、ES 很強——但 Debezium 的 TOAST value 問題、Flink PyFlink 的 JVM 交換開銷、ES 的 shard 搬遷卡住，全都發生在銜接處。

每一層的選型看起來都合理，但合理不代表簡單。合理代表「在當時的條件下，這是能力範圍內最務實的選擇」。這跟教科書上的「最佳實踐」不一樣——最佳實踐假設你有無限資源和時間，務實的選型假設你沒有。
