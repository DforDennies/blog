   2. ## **專案摘要** {#專案摘要}

   本工作說明書由 Client B（製造業）（以下簡稱甲方）委託  iKala  互動媒體股份有限公司（以下簡稱本公司或乙方）執行本案，旨在透過生成式 AI 技術編寫產品線 F 系列產品的 User Manual 使用者手冊文檔 ，以支持甲方內部將能夠有效地評估並優化其內部 AI 導入策略，從而實現更高效、更具競爭力的業務運營。

   由甲方專案團隊負責訓練資料提供、使用者需求訪談、開發成果驗收及甲方內部技術移轉作業，乙方專案團隊則負責專案系統設計分析與設計、模型訓練開發專案系統設計分析與設計、模型訓練開發與技術移轉之工作範圍。甲方相關人員共同致力於本系統之規劃與建置工作，並提此工作說明書做為專案執行之參考依據。

2. # **功能說明** {#功能說明}

   1. ## **AI User Manual 模型及資料處理流程** {#ai-user-manual-模型及資料處理流程}

   本專案將使用甲方提供工業電腦系列的產品 User Manual PDF, Word 檔案（建議 Word 檔案)，經由  iKala  將檔案清洗與整理轉為資料集，與使用者當次上傳的參考檔案，作為以下 工業電腦系列的模型運算處理回覆。

   透過AI訓練模型建構兩級層次進行，第一層為固定底層，該層級將納入工業電腦系列（產品線 A~L）近 5 年內所有的產品線之 60％的 User Manual 檔案，且不包含本案驗收對照用所使用之資料檔案。資料檔案進行訓練，以保留模型泛化能力；第二層為目標層模型，基於底層模型訓練，使其成為產品線 F 系列工業電腦專精的模型。透過上傳專案範疇之檔案，搭配系統在向量資料庫資料庫中所擁有的工業電腦系列數據，作為使用者要建立新 User Manual 輸入基礎數據。根據本次目標 User Manual 資訊 ( 產品線、產品規格等) 比對向量資料庫的資訊，Parsing 出相似度高的結果，並以此建立新的 User Manual。另須注意 AI 生成之 User Manual 非 100% 正確，其目標是提升 User Manual 產出效率。

   

   詳細執行方法請參考以下說明：

1. **Pre-processing**  
   透過模型訓練完成後可選定既有模型，將兩類型的文件進行預處理：  
   * 建立 Vector Database  
     * 為建立 RAG 檢索使用  
   * 辨識文章結構 (Structure \= TOC)  
     * 生成 User Manual 的目錄頁 TOC（Table of Contents）及 Chapter 1\. Overview (參考下圖三)  
       [圖三已移除]

                             圖三、User Manual 檔案目錄 (Table of Contents)

* Metadata (spec/components)  
  * 本次專案 AI 將辨識 User Manual 架構，並透過雙方共同標註章節關鍵內容作為 Metadata  
    * 透過監督式學習訓練資料，使模型準確找到文章段落的對應 Metadata 利於新 User Manual 生成參考依據。

2. **Retrieval**

2. ## **AI模型訓練資料說明** {#ai模型訓練資料說明}

   本專案需請甲方提供第一層固定底層所需的工業電腦系列（產品線 A~L）近 5 年內所有的產品線之 60％的 User Manual 檔案，且不包含本案驗收對照用所使用之資料檔案給予  iKala ，並由  iKala  將檔案清洗與整理轉為資料集，以利後續模型訓練，其他系列則不包含於本案。以下為本專案訓練資料之限制：

| 項目 | 說明 |
| :---- | :---- |
| 不支持 image 格式 | 在解析文件上，不需要支持圖片解析 |
| 目標層級 | 由甲方提供工業電腦產品線 F 系列產品的 User Manual PDF, Word 檔案，其數量需不少於50份不同產品。如此批訓練資料經 AI Lab 判斷不合適，  iKala  可由官網Support Portal 查詢下載更合適的檔案，並藉此再生成 100 份資料。 |
| 固定底層級 | 由甲方提供工業電腦系列（產品線 A~L）近 5 年內所有的產品線之 60％的 User Manual 檔案，作為該層級的訓練資料，並藉此作為泛化範圍鎖定。 |
| Protocol | 遵守 RESTful API 風格，並使用 JSON 格式溝通 （JSON ） |

| 安全性 | 使用在內部網路，不需要有 Auth 、 Token |

* **AI 模型驗收標準**  
  * AI 模型驗收指標說明  
    * Pre-Processing 指標：  
      1. 採用 F1-Score 是一個結合了模型的精度和召回率的綜合指標，它能夠全面評估模型在任務中的表現。  
      2. F1-Score \= 2 \* Precision \* Recall / (Precision \+ Recall)。

			[圖已移除]  
*Precision \= True-Positive / (True-Positive \+ False-Negative)*  
*Recall \= True-Positive / (True-Positive \+ False-Positive)*

* Retrieval 指標：  
  採用 Mean Reciprocal Rank (MRR)，用於衡量搜尋引擎、問答系統等在回覆結果的準確性與效率。  
  1. Mean Reciprocal Rank \= 1/U * sum(1/rank_i)

     *其中，N是查詢的數量，rank i 是第 i 個查詢的第一個相關結果的排名位置。*

     2. MRR 通常用於評估資訊檢索系統的性能，特別是在需要返回有序結果列表的任務中，例如搜尋引擎中的網頁排名，問答系統中的答案排名等。  
     * 質化評估 指標  
       1. 讓評估者對生成的文本進行打分，通常使用 1 \~ 7 的評分量表，該評分量表將由  iKala  提供初版，並經由與甲方討論並決定量表內容後由甲方決定最後定案  
       2. 自行定義評估標準，如正確性、完整性、可讀性、一致性等  
       3. 要求多位評估者對每個生成的文本按照這些標準進行評分  
       4. 計算每個標準的平均分，並分析分數的分佈情況

  * AI 模型驗收基準：  
    * F1-Score 基準 :   
      1. 第一階段交付 ：F1-Score \= 0.5。  
      2. 第二階段交付：F1-Score \= 0.6 \~ 0.7 之間。  
    * Mean Reciprocal Rank (MRR) 交付基準：  
      1. 第一階段交付：不採用 MRR 作為交付基礎。  
      2. 第二階段交付：MRR \>= 0.3。  
         *本次專案預期達到每 3 個推薦裡會有 1 個相關，所以設定 MRR 分數 0.3 為標準。*  
* Likert 評分量表 基準：  
  * 1\~7分，平均後的分數 \>=4.5
